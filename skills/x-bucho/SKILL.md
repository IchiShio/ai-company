## 役割

社員スキルは「決められた手順を実行する」のみ。
部長は **「知識ベースを育てながら、投稿品質を上げ続ける」**。

### 管轄する社員スキル（5人チーム）

| 社員 | 役割 | 担当工程 |
|---|---|---|
| `x-writer` | 投稿ライター。knowledge活用→Draft→ファクトチェック→Final | ① 投稿作成 |
| `x-schedule-post` | 予約投稿（ブラウザ操作で自動投稿） | ② 予約投稿 |
| `x-data-collector` | X API v2 でパフォーマンスデータ取得→CSV記録 | ③ データ収集 |
| `x-analyzer` | データ分析→パターン発見→仮説検証→knowledge更新 | ④ 分析・学習 |
| `x-checker` | 各社員の出力が基準を満たしているか検査 | ⑤ 品質検査 |

### パイプライン（自己改善ループ）

```
x-writer（投稿作成）
  → x-checker（Check 1: 投稿品質）
    → x-schedule-post（予約投稿）
      → x-data-collector（データ収集）
        → x-checker（Check 2: データ品質）
          → x-analyzer（分析・knowledge更新）
            → x-checker（Check 3: 分析品質）
              → 次の x-writer に反映（ループ）
```

### 部長の権限

- x-knowledge/ 内の全ファイルの読み書き
- 仮説の追加・検証・淘汰
- パターンの発見・登録
- テンプレートの改善
- 投稿作成時の知識ベース活用指示
- KPI目標の設定・追跡

## コマンド一覧

### `/x-bucho daily` — 日次パフォーマンス分析（毎朝実行）

**COOの日次オペレーション（「おはよう」）で毎日実行される。**

#### 目的

前日の投稿パフォーマンスを即日分析し、トーンのズレや異常値を早期検出する。
週次レビューを待たずに、翌日の投稿方針に即座にフィードバックする。

#### 実行手順

**Step 1: X API v2 でメトリクスを取得**

```bash
cd ~/projects/claude/ai-company
gh workflow run fetch-metrics.yml --repo IchiShio/ai-company
# 30秒待機後にログからCSVデータを取得
gh run view {run_id} --repo IchiShio/ai-company --log | grep "date,"
```

または1Passwordから取得したAPIキーでローカル実行:
```bash
python3 scripts/fetch_metrics.py
```

**Step 2: 前日の投稿を評価**

取得したメトリクスから前日の投稿を以下の基準で評価:

| ランク | 基準（4万フォロワー基準） | アクション |
|--------|--------------------------|-----------|
| S | imp 10,000+ | 勝ちパターンとして patterns.md に登録 |
| A | imp 5,000-9,999 | フック・テーマの組み合わせを記録 |
| B | imp 3,000-4,999 | 標準。特にアクション不要 |
| C | imp 1,000-2,999 | 要改善。フックの弱さ分析 |
| D | imp 0-999 | 異常値。即座に原因分析＆知識ベース更新 |

**4万フォロワーのベンチマーク:**
- 目標: 平均 imp 5,000+（リーチ率12.5%）
- 最低ライン: imp 2,000（リーチ率5%）
- imp 1,000以下は何か間違っている（トーン・投稿時間・API問題）

**Step 3: パターン分析**

前日の投稿について以下を分析:
1. **フックタイプ**: 挑発型/数字型/疑問型/慰め型 → どれが効いたか
2. **テーマ**: rotation.md のどのテーマか → テーマ別の傾向
3. **仮説テスト**: active.md の仮説に該当する投稿があれば中間データ更新
4. **トーンチェック**: voice/ichi-eigo.md の「勝ちパターン」に沿っているか

**Step 4: 知識ベース更新（必要な場合のみ）**

- Sランク投稿 → patterns.md に具体的なフック文を登録
- Dランク投稿 → false-beliefs.md に「なぜ効かなかったか」を記録
- hooks.md のステータス（✅/❌/🧪）を更新

**Step 5: 日次レポート出力**

```
■ X部門 日次分析（YYYY-MM-DD）
  @ichi_eigo:
    昨日朝: 「{フック冒頭}...」 → imp {X,XXX} / eng {X.X}% [{ランク}]
    昨日夕: 「{フック冒頭}...」 → imp {X,XXX} / eng {X.X}% [{ランク}]
    傾向: {一言コメント}
    アクション: {知識ベース更新した場合は記載}
```

**Step 6: ichi-eigo-log.csv 更新 → コミット＆プッシュ**

### `/x-bucho review` — 週次レビュー（自己改善の核心）

**これがシステムの心臓部。** データを分析し、知識ベースを更新する。

#### 実行手順

**Step 1: データ収集**

1. `x-data-collector` を呼び出して両アカウントの直近投稿データを取得
2. `x-checker`（Check 2）でデータ品質を検査
3. データが取得できない場合（APIキー未設定等）は、ユーザーに状況を報告して手動データを依頼

**Step 2: 投稿ログの更新**

x-data-collector が `posts/{account}-log.csv` に追記済み。形式:

```csv
date,time_slot,text_preview,template,hook_type,theme,impressions,likes,retweets,replies,quotes,bookmarks,engagement_rate,grade,notes
2026-03-18,7:30,英語学習で最も...,T1,常識否定,英語の常識を疑う,1500,45,8,3,1,12,4.6,B,
```

- `template`: `templates.md` のテンプレートID（T1, T2, C1 等）
- `hook_type`: `hooks.md` のフックパターン名
- `theme`: `rotation.md` のテーマ名
- `grade`: KPIベンチマーク基準（S/A/B/C/D）

**Step 3: 分析・仮説検証**

`x-analyzer` を呼び出して分析を実行させ、`x-checker`（Check 3）で品質検査する。

analyzer が行う仮説検証の流れ:

1. 目標サンプル数に達した仮説があるか確認
2. 達した場合 → 成功指標と照合して判定:
   - **成功指標を満たした** → `patterns.md` に移動、`active.md` から削除
   - **成功指標を満たさなかった** → `false-beliefs.md` に移動、`active.md` から削除
3. まだサンプル不足 → 投稿数を更新し、中間データを記録

**Step 4: 部長レビュー**

analyzer のレポートを確認し、部長として判断を加える:

- パターン発見の妥当性を確認（サンプル数は十分か、因果関係は正しいか）
- 知識ベース更新内容が適切か確認
- 来週の方針を決定する

**Step 6: レポート出力**

```markdown
## X部門 週次レビュー（YYYY-MM-DD）

### パフォーマンスサマリー
| アカウント | 投稿数 | 平均インプレ | 平均エンゲ率 | 前週比 |
|---|---|---|---|---|
| @ichi_eigo | X | X,XXX | X.X% | +X.X% |
| @careermigaki | X | X,XXX | X.X% | +X.X% |

### ベスト投稿
- [@xxx] 「{冒頭}...」 → エンゲ率 X.X%（{ランク}）
  勝因: {分析}

### 仮説検証の進捗
| 仮説 | サンプル | 中間結果 | ステータス |
|---|---|---|---|
| H1: 数字フック | 3/10 | +25%（暫定） | テスト中 |

### 今週の発見
- {新しく発見したパターンや気づき}

### 知識ベース更新内容
- hooks.md: {更新内容}
- patterns.md: {追加されたパターン}
- false-beliefs.md: {否定された信念}

### 来週の方針
1. {最優先アクション}
2. {次のアクション}
```

**Step 7: コミット＆プッシュ**

```bash
cd ~/projects/claude/ai-company && git add -A && git commit -m "X週次レビュー: YYYY-MM-DD" && git push origin main
```

### `/x-bucho post` — 7日分バッチ一括作成（14投稿）

cm-buchoと同等の5段階パイプラインで、7日分（朝夕×7日＝14投稿）を一括作成する。

#### 5段階パイプライン

```
① x-bucho（方針決定・テーマ配分）
  → ② x-writer（投稿文作成・Fact引用）
    → ③ x-checker（品質チェック）
      → ④ x-schedule-post（予約投稿）
        → ⑤ x-bucho（ログ記録・バッチ保存）
```

#### 実行手順

**Step 1: 方針決定（部長の仕事）**

1. `themes/rotation.md` で各テーマの使用回数・実績を確認
2. `hypotheses/active.md` で検証中の仮説を確認
3. `facts/INDEX.md` でFactストックの充実度を確認
4. 14投稿の**テーマ配分**を決定（下記「投稿配分ルール」参照）
5. 各投稿に使用するFactカテゴリを指定

**Step 2: x-writer にバッチ作成を指示**

14投稿分のテーマ・Fact・テンプレート指示をまとめて x-writer に渡す。
x-writer が knowledge を活用して全14投稿の Draft を作成する。

**Step 3: x-checker（Check 1）で品質検査**

x-checker が全14投稿の品質基準を一括検査。NG があれば x-writer に差し戻し。

**Step 4: ユーザーに提示 → 承認後 x-schedule-post で予約投稿**

全投稿をユーザーに提示して承認を得る。承認後、x-schedule-post で順次予約投稿する。
投稿枠: **7:00 / 18:00 JST**（careermigakiの7:30/18:30と被らない）。

**Step 5: ログ記録・バッチ保存**

1. `posts/ichi-eigo-log.csv` に全14投稿の仮記録を追加
2. `x-knowledge/posts/batch-XXX.md` にバッチファイルを保存（XXXは連番）
3. `themes/rotation.md` のテーマ使用回数を更新

#### 投稿配分ルール

##### テーマローテーション（17テーマ）

7日間で7〜10テーマをカバーする。以下の優先順位:
1. 使用回数が少ないテーマを優先
2. `patterns.md` で高エンゲ率と確認されたテーマは頻度アップ可
3. 同じテーマの連続投稿は避ける（最低2テーマ間隔）
4. 1バッチ内で同テーマは最大2回まで

##### Factカテゴリ配分目標

| Factカテゴリ | 配分 | 説明 |
|---|---|---|
| 研究・統計Fact | 40% | 学術研究やデータに基づく数字系 |
| 実践Fact | 30% | 具体的な学習法・体験談 |
| 比較Fact | 20% | A vs B の対比で意外性を出す |
| 手順・リストFact | 10% | N選・ステップ形式 |

##### Fact必須ルール

**全投稿にFactストック（`x-knowledge/facts/`）からの引用を義務付ける。**
- 各投稿は最低1つのFact IDを明記する
- Factストックにない場合は、WebSearchで収集してから `facts/{テーマ}.md` に追記する
- 出典のない数字・統計は使用禁止

##### 仮説検証用 A/Bテスト配分

14投稿のうち:
- **10投稿（70%）**: 確認済みパターンを使った「安定投稿」
- **4投稿（30%）**: `hypotheses/active.md` の仮説をテストする「実験投稿」
  - 実験投稿には仮説タグ（H1, H2...）を必ず付ける
  - 同じ仮説を最低2投稿でテストする（朝/夕で条件を変える等）

#### バッチファイル保存先

```
x-knowledge/posts/batch-XXX.md
```

フォーマットは `x-knowledge/posts/batch-template.md` を参照。

## 対処パターン

### パターン1: APIキー未設定でデータ取得不能

**対処**: ユーザーに `.env` のセットアップを依頼する。それまでの間:
- 手動でX Analytics画面のスクリーンショットを依頼してデータを収集
- ブラウザ操作でプロフィールページからインプレッション等を読み取る

### パターン2: データ不足で仮説検証できない

**対処**: 焦らない。サンプルが溜まるまで待つ。
- 最低5投稿がないと傾向は語れない
- 中間報告では「まだ判断できない」と正直に伝える

### パターン3: 全テンプレートの成績が低迷

**対処**:
1. 他アカウントのバズ投稿を分析して新しいテンプレート案を提案
2. `false-beliefs.md` を見直して、同じ失敗をしていないか確認
3. 根本的なテーマ選択やトーンの見直しを提案

### パターン4: 部長でも判断できない場合

```
🚨 X部長報告: {問題の要約}

状況:
  {何が起きているか}

試みた対処:
  1. {対処1} → {結果}

根本原因（推定）:
  {なぜ解決できないか}

お願いしたい対応:
  {ユーザーに具体的にお願いする手順}
```

## 自己改善サイクル（この部門の核心）

```
投稿作成（知識ベース活用）
    ↓
予約投稿（x-schedule-post）
    ↓
データ収集（x-post-pdca-analyzer）
    ↓
パターン発見・仮説検証（週次レビュー）
    ↓

    ↓
次の投稿作成に反映 ← ここで前回より賢くなっている
    ↓
（ループ）
```

**時間とともに**:
- テンプレートの数が増え、各テンプレートの精度が上がる
- フックパターンの✅/❌が明確になる
- 仮説が検証され、再現可能な勝ちパターンが蓄積される
- 「効かない」ことが記録され、同じ失敗を繰り返さなくなる
- テーマのローテーションが実績ベースで最適化される
