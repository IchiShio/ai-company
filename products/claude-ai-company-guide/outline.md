### 第1章：AI会社とは何か — 設計思想
- スキル = 社員。SKILL.mdが「職務記述書」
- なぜ1つの万能プロンプトではなく「組織」にするのか
  - 責務分離 → 品質が上がる
  - パイプライン化 → 自動化できる
  - 交換可能 → 1つのスキルだけ改善・差し替えできる
- 組織の3層構造
  - 社員スキル（実行者）: 決められた手順を実行する
  - 部長スキル（判断者）: 状況判断・代替手段・エスカレーション
  - COO（統括者）: 全社横断の優先度調整・KPI管理
- エスカレーション設計
  - 社員が失敗 → 部長が判断 → 部長でも無理 → 社長（人間）に報告
  - 人間が介入するのは「例外中の例外」だけ

---

### 第2章：最小構成で始める — 最初のスキルを作る
- SKILL.mdの基本構造（frontmatter + 本文）
- 実例：「記事ライター」スキルを作る
  ```
  スキル名: article-writer
  役割: 指定テーマの記事を執筆する
  入力: テーマ、ターゲット読者、文字数
  出力: Markdown記事
  品質基準: ...
  ```
- Claude Codeでスキルを呼び出す方法
- ファイル配置（~/.claude/skills/ へのシンボリックリンク）
- やってみよう：自分の業務で1つスキルを作る

---

### 第3章：品質ゲートを作る — チェッカースキル
- なぜチェッカーが必要か
  - AIは「やりました」と言って手を抜く（実例）
  - チェッカーがないと品質が崩壊する
- チェッカースキルの設計パターン
  - 入力: 前工程の出力
  - 検査項目リスト（明確な合格基準）
  - 判定: PASS / FAIL + 理由
  - FAIL時の挙動（差し戻し or エスカレーション）
- 実例：記事チェッカー
  - ファクトチェック、NGワード、文字数、構成確認
- 2回連続FAIL → エスカレーションのルール

---

### 第4章：パイプラインを組む — 自動で流れる仕組み
- パイプラインとは「スキルの連鎖実行」
- 3ステップパイプラインの実例
  ```
  データ収集 → 分析 → 実行
  各ステップ後にチェッカーが品質検査
  ```
- Agent tool でスキルを順次呼び出す方法
- シェルスクリプトで自動化する方法
- エラーハンドリングと中断ルール
- 実例：SEOパイプライン（native-real.comの事例）
  - Step 1: Search Console/GA4のデータを自動取得
  - Step 2: データを分析してアクションリスト生成
  - Step 3: 上位アクションを自動実行（title改善、内部リンク等）
  - 各Step後にチェッカーが検査

---

### 第5章：部長を雇う — 判断できるAIを作る
- 社員スキルと部長スキルの違い
  - 社員: 手順通りに実行
  - 部長: 状況を見て判断し、社員に指示を出す
- 部長スキルに持たせる権限
  - 社員スキルの呼び出し（Agent tool）
  - KPI確認と方針決定
  - 失敗時のリカバリー判断
  - 社長への報告（Telegram通知等）
- 部長スキルの設計パターン
  - `/bucho post` → 投稿バッチ作成を指揮
  - `/bucho review` → 週次レビュー実行
  - `/bucho kpi` → KPI確認
- 実例：X部門部長の1週間
  - 月曜: 先週のデータ分析指示 → パターン更新
  - 水曜: 今週の投稿バッチ作成指示 → 品質チェック → 予約
  - 金曜: 中間パフォーマンス確認

---

### 第6章：知識ベースを育てる — 自己改善するAI組織
- なぜ知識ベースが必要か
  - スキルだけでは「毎回ゼロから」になる
  - 過去の成功・失敗を蓄積して次に活かす
- 知識ベースの構成
  ```
  knowledge/
  ├── craft/        # テンプレート、フック、ルール
  ├── facts/        # 根拠あるデータ・統計
  ├── hypotheses/   # テスト中の仮説
  ├── learnings/    # 確認済みパターン + 否定された信念
  ├── posts/        # 過去の出力ログ + パフォーマンス
  └── INDEX.md      # ナビゲーション
  ```
- 自己改善ループの仕組み
  ```
  実行 → データ収集 → 分析 → パターン発見 → 知識更新 → 次の実行に反映
  ```
- 仮説検証の方法
  - 仮説を立てる（例：「数字を入れた投稿はimpが高い」）
  - データで検証
  - 確認 → patterns.mdに昇格 / 否定 → false-beliefs.mdに記録
- メモリシステム（MEMORY.md）との違い
  - メモリ = 人間（社長）の好みや方針
  - 知識ベース = 組織が学んだ業務知識

---

### 第7章：全社を統括する — COOと経営ダッシュボード
- COOスキルの役割
  - 複数部門のKPIを集約
  - 部門間の優先度調整
  - コスト追跡（API使用量等）
  - 社長への統合レポート
- ダッシュボードの設計
  - 各部門の今週のKPI
  - 進行中タスク
  - 異常・アラート
- cronで定期実行する設定
  - 毎朝: 日次レポート
  - 毎週月曜: 週次レビュー
- Telegram/Discord連携で外出先から経営する方法
