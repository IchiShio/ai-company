# 社員は全員AI。Claude Codeでひとり会社を自動化した全手順

---

# はじめに

> ※本記事は2026年3月時点の情報に基づいています。Claude Codeの仕様は変更される可能性がありますが、設計思想（1スキル=1責務、チェッカー、エスカレーション）は普遍的に適用できます。

## 1人で会社を回している。社員は全員AI。

僕は今、1人で「会社」を経営しています。

英語学習メディアの運営、SEO改善、X（Twitter）での毎日の情報発信、データ分析、品質チェック——これらの業務を、ほぼ人間の手を使わずに回しています。

社員は全員AIです。

「AIに仕事を手伝ってもらう」という話ではありません。組織として機能するAIの「会社」を、Claude Code上に構築した話です。

部長がいて、社員がいて、チェッカーがいて、COO（最高執行責任者）がいる。それぞれが自分の役割を理解していて、判断が必要な場面では上にエスカレーションし、人間（つまり僕）が動くのは本当に例外的なケースだけ。

この仕組みを作るのに、プログラミングの深い知識は必要ありませんでした。必要だったのは「業務を分解する力」と「Claude Codeのスキル機能の理解」、そしてちょっとした組織設計の発想です。

## 具体的に何が起きているか

少しだけ、実際の運用を見せます。

### SEOパイプラインが自動で回る

僕が運営しているWebサイトでは、SEO改善が週次で自動実行されます。

```
/seo-pipeline
```

このコマンド1つで、以下が全自動で走ります。

1. **データ収集**: Search Console、GA4、Ahrefsから12種類のCSVを自動取得
2. **分析**: 7つの分析フレームワークで評価し、「今週やるべきこと」Top 5を特定
3. **実行**: titleタグの改善、meta descriptionの最適化、内部リンクの整備を自動で実施
4. **品質検査**: 各ステップの後にチェッカーが品質を検査。基準未達なら即中断

人間がやることは、最初にコマンドを打つことだけ。あとはAIの「社員たち」が自分の仕事をこなしてくれます。

### X投稿が自動で生成・チェック・予約される

運営しているXアカウント（フォロワー数万人）の投稿も、AI社員が担当しています。

- **ライター**が投稿文を作成する
- **チェッカー**がファクトチェックと品質基準を検査する
- **スケジューラー**が空き枠を見つけて予約投稿する
- **アナライザー**がパフォーマンスを分析して、次回の改善点を知識ベースに蓄積する

部長（x-bucho）がこれらの社員を統括し、週次でパフォーマンスレビューを行って投稿パターンを改善していきます。

### データ分析が勝手に回る

毎朝6時に、SEOデータの収集が自動で走ります。Search Console、GA4、Ahrefsの3つのソースからデータを引っ張ってきて、Google Driveに日付別で保存。これが分析の材料になります。

X投稿のパフォーマンスデータも同様に自動で取得され、「どんな投稿が伸びたか」「何が滑ったか」を7つの分析軸で評価します。

## 「プロンプトを工夫する」のとは違う

ここで大事なことを言います。

これは「すごいプロンプトを書いた」という話ではありません。

1つの長いプロンプトにすべてを詰め込んで、「あれもやって、これもやって」と指示するやり方は、最初はうまくいくかもしれません。でも、業務が増えてくると必ず破綻します。品質が下がる。抜け漏れが出る。何がうまくいって何がダメだったのか、切り分けられなくなる。

僕がやったのは、AIを「組織」として設計し直すことでした。

1つの業務を1人の社員に任せる。その社員の仕事をチェックする別の社員を置く。複数の社員を束ねて判断する部長を置く。部門間の調整をするCOOを置く。

人間の会社と同じ構造です。

そしてこの構造は、Claude Codeの「スキル」という仕組みを使えば、驚くほどシンプルに実現できます。

# 第1章：AI会社とは何か — 設計思想

## スキル = 社員、SKILL.md = 職務記述書

Claude Codeには「スキル」という仕組みがあります。

`~/.claude/skills/` というフォルダの中に、`SKILL.md` というMarkdownファイルを置く。たったこれだけで、Claude Codeに「新しい能力」を追加できます。

```
~/.claude/skills/
├── seo-data-collector/
│   └── SKILL.md    ← SEOデータを自動収集する「社員」
├── seo-analyzer/
│   └── SKILL.md    ← データを分析して改善案を出す「社員」
├── seo-checker/
│   └── SKILL.md    ← 品質を検査する「社員」
└── x-writer/
    └── SKILL.md    ← X投稿を書く「社員」
```

ここで発想を転換します。

**スキルを「便利なテンプレート」ではなく「社員」として設計する。**

SKILL.mdは単なるプロンプトのテンプレートではありません。**職務記述書（Job Description）** です。この社員は何を担当し、何を入力として受け取り、何を出力し、どんな品質基準で仕事をし、困ったときに誰に相談するのか——それを明確に書いたドキュメントです。

実際のSKILL.mdの構造を見てみましょう。

```yaml
---
name: seo-data-collector
description: |
  自社サイトのSEOデータを3つのソースから自動収集し、
  Google Driveに日付別CSVとして保存する。データ収集のみ。
  分析は別スキルが担当。
---
```

注目してほしいのは、最後の一文です。「**分析は別スキルが担当。**」

この一文が、「万能プロンプト」と「組織」を分ける境界線です。

---

## なぜ1つの万能プロンプトではダメなのか

最初に思いつくのは、こういうアプローチです。

> 「SEOデータを収集して、分析して、改善施策を考えて、実行して、結果をチェックして、問題があったら直して、レポートにまとめて。」

1つのプロンプトに全部詰め込む。最初のうちは、これでも動きます。Claudeは賢いので、なんとなくいい感じにやってくれます。

でも、これが破綻する理由は3つあります。

### 理由1：責務が曖昧になると、品質が下がる

人間の会社でも同じです。「営業も経理も採用も全部やって」と言われた社員は、すべてが中途半端になります。

AIも同じで、1つのプロンプトに5つの仕事を詰め込むと、どの仕事にも100%の注意を払えなくなります。データ収集で手を抜いたり、分析が浅くなったり、チェックが甘くなったり。

**1スキル = 1責務**にすると、そのスキルは自分の仕事だけに集中できます。データ収集スキルは、データの取得漏れがないことだけを気にすればいい。分析スキルは、7つのフレームワークで深く分析することだけに集中できる。

### 理由2：パイプライン化できない

全部を1つのプロンプトでやると、途中で止められません。

「データ収集まではOKだけど、分析の品質がイマイチだからやり直して」ということができない。全部をもう一度最初からやり直すしかない。

スキルを分けておけば、パイプラインとして組めます。

```
データ収集 → [品質チェック] → 分析 → [品質チェック] → 実行 → [品質チェック]
```

各ステップの間にチェックポイントを置ける。問題があったステップだけやり直せる。これは1つのプロンプトでは絶対にできないことです。

### 理由3：改善・差し替えができない

万能プロンプトを改善しようとすると、全体を見直す必要があります。データ収集の部分を直したら、分析の部分に影響が出た——こういうことが頻繁に起きます。

スキルを分けておけば、1つのスキルだけを改善・差し替えできます。「分析の精度が低いから、analyzerのSKILL.mdだけ書き直そう」ということが可能になります。他のスキルには一切影響しません。

ソフトウェア開発の世界では、これを「疎結合」と呼びます。難しい概念ではありません。**それぞれの部品が独立しているから、1つだけ直せる**——ただそれだけのことです。

## 組織の3層構造

AI会社の組織は3つの層で構成されます。

```
┌─────────────────────────────────────────┐
│              社長（人間＝あなた）              │
│    最終判断 / 方針決定 / 例外対応           │
└────────────────────┬────────────────────┘
                     │ エスカレーション
                     │ （部長でも解決できないとき）
┌────────────────────▼────────────────────┐
│            COO（最高執行責任者）              │
│    部門横断の調整 / KPI管理 / 統合レポート    │
└───────┬─────────────────────┬───────────┘
        │                     │
┌───────▼───────┐     ┌───────▼───────┐
│   部長（判断者）  │     │   部長（判断者）  │
│   SEO部門       │     │    X部門        │
│  状況判断・指示   │     │  状況判断・指示   │
└───┬───┬───┬───┘     └───┬───┬───┬───┘
    │   │   │             │   │   │
   社員 社員 社員          社員 社員 社員
 collector              writer
 analyzer              checker
 executor              scheduler
 checker               analyzer
```

それぞれの役割を見ていきましょう。

### 第1層：社員スキル（実行者）

社員スキルは、決められた手順を忠実に実行します。

- **collector**: データを集める
- **analyzer**: データを分析する
- **executor**: 改善施策を実行する
- **writer**: 文章を書く
- **checker**: 品質を検査する
- **scheduler**: 予約投稿する

社員の特徴は「判断しない」ことです。SKILL.mdに書かれた手順に従って、与えられた仕事をきっちりこなす。想定外のことが起きたら、自分で判断せずに上（チェッカーや部長）に報告します。

### 第2層：部長スキル（判断者）

部長スキルは、社員を束ねて判断します。

僕の場合、以下の2人の部長がいます。

- **seo-bucho**: SEO部門の部長。collector、analyzer、executor、checkerを指揮する
- **x-bucho**: X部門の部長。writer、checker、scheduler、analyzerを指揮する

部長の仕事は3つです。

**1. 社員への指示出し**

部長は状況を見て、どの社員にどんな仕事を振るかを判断します。

```
「今週のデータを見て、CTRが低いページが5つある。
analyzer、このデータでTop 5アクションを出してくれ。」
```

**2. 失敗時のリカバリー判断**

チェッカーが「FAIL」を出したとき、部長が対処方法を判断します。

```
「analyzerの分析品質が不足しているな。
具体的なページ名と数値を入れて再分析するよう指示し直そう。」
```

**3. 解決不能時のエスカレーション**

部長でも解決できない問題だけを、社長（人間）に報告します。

```
「Search Console APIが3日連続でエラーを返している。
APIの認証情報の更新が必要かもしれません。社長、確認をお願いします。」
```

### 第3層：COO（統括者）

COOは複数の部門を横断的に管理します。

- 各部門のKPIを集約して全社ダッシュボードを作る
- 部門間の優先度を調整する（「今週はSEOより投稿を優先」等）
- API使用量やコストを追跡する
- 社長への統合レポートを作成する

小さく始めるなら、最初はCOOは不要です。部門が2つ以上になったときに初めて必要になります。

## エスカレーション設計——人間が動くのは「例外中の例外」

この組織で最も重要な設計が「エスカレーション」です。

問題が起きたとき、以下の順序で解決を試みます。

```
Step 1: 社員が自分で対処を試みる
   ↓ （失敗）
Step 2: チェッカーが検知して差し戻す
   ↓ （2回連続FAIL）
Step 3: 部長がリカバリー方法を判断する
   ↓ （部長でも解決できない）
Step 4: 社長（人間）に報告する
```

具体例を見てみましょう。

### 例：SEOパイプラインでの障害

```
1. collector がGA4のデータ取得に失敗した
   → collector が3回リトライする（社員レベルの自動リカバリー）

2. 3回とも失敗した
   → checker が「❌ FAIL: GA4データ欠損」を報告
   → パイプラインが中断される

3. 部長（seo-bucho）がエスカレーションを受け取る
   → 「GA4なしでもSearch ConsoleとAhrefsのデータで分析できるか？」を判断
   → 可能と判断 → analyzerに「GA4抜きで分析続行」を指示

4. もし部長でも対処不能な場合（APIの認証切れ等）
   → 社長に構造化レポートで報告
   → 「GA4 API認証が切れています。再認証が必要です。」
```

ポイントは、**Step 4に到達することはほとんどない**ということです。

大抵の問題はStep 1〜3で解決します。人間である僕が介入するのは、APIキーの更新やサービスの契約変更など、物理的にAIには不可能な作業だけです。

### 「2回連続FAILルール」

チェッカーの設計で特に重要なのが「2回連続FAIL」のルールです。

```
1回目のFAIL → 社員に差し戻してやり直させる
2回目のFAIL → 同じ問題が再発した → 社員レベルでは解決不能と判断
             → 部長にエスカレーション
```

これがないと、社員とチェッカーが永遠にFAIL→やり直し→FAILのループに入ります。人間の組織でも「2回やってダメなら上に相談しろ」というルールがありますよね。それと同じです。

## なぜ「組織」なのか——メンタルモデルの力

ここまで読んで、「これって要するにマイクロサービスアーキテクチャじゃないの？」と思ったエンジニアの方もいるかもしれません。

技術的には近いです。でも「組織」というメタファーを使うことには大きな意味があります。

### 設計が直感的になる

「このAPIレスポンスをパースして次のマイクロサービスに渡す」と考えるより、「この社員が仕事を終えたら、次の社員に引き継ぐ」と考えるほうが、圧倒的に設計しやすい。

「エラーハンドリングをどう設計するか」より、「この社員が困ったとき、誰に相談させるか」と考えたほうが、自然に良い設計が生まれます。

### AIへの指示が明確になる

SKILL.mdに「あなたはSEOデータを収集する社員です。分析は別の社員が担当するので、あなたはデータの正確な取得だけに集中してください」と書く。

これだけで、AIは自分の役割を正確に理解します。余計なことをしない。手を抜かない。なぜなら、自分の仕事の範囲が明確だからです。

### スケールしやすい

新しい業務が増えたら、新しい社員（スキル）を雇えばいい。チームが大きくなったら、部長を増やせばいい。とてもシンプルです。

実際、僕の組織は最初は3人の社員から始まりました。

```
初期: collector → analyzer → executor（3人）
```

今では25個以上のスキルが稼働しています。でも設計の基本は変わっていません。「1スキル = 1責務」「チェッカーで品質を保証」「困ったらエスカレーション」——この3つだけです。

---

## まとめ：AI会社の4原則

この章で解説した設計思想を、4つの原則にまとめます。

### 原則1：1スキル = 1責務
1つのスキルには1つの仕事だけを任せる。複数の仕事を詰め込まない。

### 原則2：品質はチェッカーが保証する
実行者と検査者を分ける。自分で自分の仕事をチェックさせない。

### 原則3：判断は上位層に委ねる
社員は実行に集中し、判断が必要な場面では部長に委ねる。部長でもダメなら社長に。

### 原則4：人間は例外処理だけ
日常業務はAI組織が回す。人間が動くのは、AIには物理的に不可能な作業だけ。

# 第2章：最小構成で始める — 最初のスキルを作る

前章で「AI会社」の設計思想を理解しました。ここからはいよいよ実践です。

最初にやることは、**たった1つのスキルを作る**こと。いきなり組織全体を設計する必要はありません。まずは「社員1人を雇う」ところから始めましょう。

---

## SKILL.mdとは何か

Claude Codeのスキルとは、`~/.claude/skills/` に配置された **SKILL.md** ファイルのことです。

普通のプロンプトとの違いは明確です。プロンプトは「1回きりの指示」ですが、SKILL.mdは**職務記述書**です。名前があり、役割があり、手順が定義されている。何度呼び出しても同じ品質で仕事をしてくれる「再現可能な社員」を作るのがSKILL.mdの目的です。

---

## SKILL.mdの基本構造

SKILL.mdは大きく2つのパーツで構成されています。

### 1. frontmatter（ヘッダー情報）

ファイルの先頭に `---` で囲まれたYAML形式のメタデータを書きます。

```markdown
---
name: article-writer
description: |
  ブログ記事のライター。指定テーマで記事を執筆する。

  次のような依頼で使用すること：
  - 「記事を書いて」「ブログ記事を作成して」
  - 「○○について記事にして」
---
```

**name** はスキルの識別子です。フォルダ名と一致させるのがおすすめです。

**description** は2つの役割を持っています。

1つ目は「このスキルが何をするか」の説明。2つ目が重要で、**「どんな依頼のときにこのスキルを呼び出すべきか」の判定基準**です。Claude Codeは `description` を読んで、ユーザーの依頼にどのスキルが適切かを判断します。だから「次のような依頼で使用すること」というリストを入れておくと、意図通りに呼び出されやすくなります。

### 2. 本文（職務記述書）

frontmatterの下に、Markdown形式で仕事の手順を書きます。ここが「社員への業務マニュアル」です。

```markdown
# 記事ライター

## 役割
指定されたテーマについて、ターゲット読者に合わせた記事を執筆する。

## 入力
呼び出し時に以下を受け取る：
- テーマ（必須）
- ターゲット読者（任意、デフォルト: 一般読者）
- 文字数目安（任意、デフォルト: 3,000文字）

## Step 1: リサーチ
...

## Step 2: 構成作成
...
```

ポイントは、**曖昧さを排除する**ことです。

「いい感じに書いて」ではなく、「Step 1で何をして、Step 2で何をして、最終的にどういう形式で出力するか」を明確に書く。人間の新入社員に渡す業務マニュアルを想像してください。そのレベルの具体性が必要です。

### ステップ1：フォルダを作る

### ステップ2：SKILL.mdを書く

```markdown

# 記事ライター

## 役割

指定されたテーマについて、ターゲット読者に最適化された記事を執筆する。
自分では公開しない（完成した記事をユーザーまたは後続のチェッカーに渡す）。

---

## 入力

呼び出し時に以下の情報を受け取る：

| 項目 | 必須/任意 | デフォルト |
|------|-----------|------------|
| テーマ | 必須 | — |
| ターゲット読者 | 任意 | 一般読者 |
| 文字数目安 | 任意 | 3,000文字 |
| トーン | 任意 | です・ます調 |
| 出力形式 | 任意 | Markdown |

---

## Step 1: リサーチ

テーマに関する情報を収集する。

1. 指定されたテーマのキーワードで検索し、上位記事の構成を確認する
2. 読者が知りたいであろう疑問点を3〜5個リストアップする
3. 記事に盛り込むべきデータ・事例を収集する

---

## Step 2: 構成作成

記事の骨格を作る。

1. タイトル案（2〜3案）
2. リード文（100〜150文字）
3. H2見出しの構成（5〜7セクション）
4. 各セクションで書く内容のメモ（箇条書き）

---

## Step 3: 執筆

構成に沿って本文を書く。

### 品質基準
- 文字数: 指定文字数の±10%以内
- 各H2セクション: 最低300文字
- 具体例・データ: 各セクションに最低1つ
- 読みやすさ: 1文50文字以内を目安

### 禁止事項
- 根拠のない数字（「9割の人が〜」等）
- 他サイトのコピー
- 事実と異なる主張

---

## Step 4: セルフチェック

提出前に以下を自己確認する。

- [ ] 文字数は基準範囲内か
- [ ] タイトルとリード文は内容を正確に反映しているか
- [ ] 論理の飛躍がないか
- [ ] 誤字脱字がないか
- [ ] 禁止事項に違反していないか

---

## 出力フォーマット

```
📝 記事（テーマ: {テーマ名}）

文字数: {実際の文字数}
構成: H2 × {セクション数}

---
{記事本文（Markdown形式）}
---

セルフチェック結果:
  文字数: ✅ / ❌
  具体例: ✅ / ❌
  禁止事項: ✅ / ❌
```

- チェッカースキルが存在する場合は、チェッカーの判定を待つ
- 不明な点があれば、作業を進める前にユーザーに確認する
```

### ステップ3：Claude Codeで呼び出す

## SKILL.mdを書くときの5つのコツ

実際に何十ものスキルを運用してきた経験から、効果的なSKILL.mdを書くコツをまとめます。

### 1. Stepを明示する

「いい感じにやって」ではAIは手を抜きます。Step 1、Step 2...と明確に段階を分けることで、各ステップを確実に実行させられます。

### 2. 出力フォーマットを固定する

出力の形式が毎回違うと、後続のチェッカーやパイプラインが処理できません。テンプレートを定義して、常に同じ形式で出力させましょう。

### 3. 「禁止事項」を書く

AIは指示されていないことを勝手にやりがちです。「やるべきこと」だけでなく「やってはいけないこと」を明記するのが非常に効果的です。

### 4. descriptionに呼び出し条件を書く

frontmatterの `description` に「次のような依頼で使用すること」と具体例を列挙しておくと、意図しないスキルが呼ばれる問題を防げます。

### 5. 1スキル1責務を徹底する

「記事を書いてチェックもして公開もする」を1つのスキルに詰め込まないでください。書く人、チェックする人、公開する人は別のスキルにする。これが品質を保つ最大の秘訣です。

2. その手順をStep 1, 2, 3...と書き出す
3. SKILL.mdのテンプレートに当てはめる
4. `~/.claude/skills/` に配置して呼び出してみる

最初のスキルが動いた瞬間、「これは社員だ」と実感するはずです。

ただし、ここで1つ重大な問題があります。作ったスキルに仕事をさせてみると、**AIは「やりました」と報告しておきながら、実は手を抜いている**ことがあるのです。

次の章では、この問題を解決する「チェッカースキル」を作ります。

# 第3章：品質ゲートを作る — チェッカースキル

前章で最初のスキルを作りました。記事ライターに仕事を頼むと、それらしい記事が返ってくる。

「おお、これはすごい！」

...と最初は思います。でも、しばらく運用すると気づくんです。

**AIは、平気でサボります。**

---

## AIが「やりました」と嘘をつく実例

これは実際に起きたことです。

ある記事ライタースキルに「3,000文字以上の記事を書いて」と依頼しました。返ってきた記事には「文字数: 3,200文字」と書いてありました。

実際に数えてみたら **1,800文字** でした。

別のケースでは、「データを収集して10件以上のCSVを作成して」と依頼。「✅ 完了しました。12件のデータを収集しました」と報告が来ました。

CSVを開いたら **3件** しかありませんでした。

これはAIが悪意を持って嘘をついているわけではありません。LLMの特性として、「期待されている回答を生成しようとする」傾向があるだけです。でも、結果として品質は崩壊します。

人間の組織でも同じですよね。レポートを出した本人に「このレポート、ちゃんとできてる？」と聞いたら「はい、大丈夫です」と言うに決まっています。だから組織には**レビュアー**が必要なんです。

AI会社でも同じ。**書いた人とチェックする人は、必ず分ける。**

これがチェッカースキルの存在理由です。

---

## チェッカースキルの設計パターン

チェッカースキルの構造はシンプルです。

```
入力: 前工程のスキルが出力したもの
  ↓
検査: 明確な基準に照らして1項目ずつチェック
  ↓
判定: PASS または FAIL + 具体的な理由
  ↓
FAIL時: 差し戻し or エスカレーション
```

重要なのは、**チェック項目が「客観的で曖昧さがない」こと**です。

ダメな例：
- ❌「記事の品質が高いか」→ 何をもって「高い」とするの？
- ❌「読みやすいか」→ 主観的すぎて判定できない

良い例：
- ✅「文字数が3,000文字以上か」→ 数えれば分かる
- ✅「H2見出しが5個以上あるか」→ 数えれば分かる
- ✅「根拠のない数字（9割、99%等）を含んでいないか」→ 検索すれば分かる

チェッカーに主観的な判断をさせると、チェッカー自身が「まあ大丈夫でしょう」と甘く通してしまいます。**数えられるもの、検索できるもの、照合できるもの**だけをチェック項目にするのがコツです。

---

## 実例：記事チェッカーを作る

前章の記事ライターに対応するチェッカーを作りましょう。

```markdown
---
name: article-checker
description: |
  記事ライターの出力が品質基準を満たしているか検査するスキル。
  NGなら差し戻し、2回連続NGならユーザーにエスカレーションする。

  次のような依頼で使用すること：
  - 「記事をチェックして」「品質を検査して」
  - パイプラインの品質ゲートとして自動呼び出し
---

# 記事チェッカー

## 役割

記事ライター（article-writer）の出力が品質基準を満たしているか検査する。
基準未達ならリトライを指示し、2回連続NGならユーザーにエスカレーションする。

---

## 検査項目

| # | チェック項目 | 合格基準 | 重大度 |
|---|---|---|---|
| 1 | 文字数 | 指定文字数の±10%以内 | FAIL |
| 2 | H2見出し数 | 5個以上 | FAIL |
| 3 | 各セクション文字数 | 各H2セクション300文字以上 | WARNING |
| 4 | 具体例・データ | 各セクションに最低1つ | WARNING |
| 5 | 根拠なし数字 | 「9割」「99%」等の出典なし数字がない | FAIL |
| 6 | タイトルとの整合性 | タイトルの約束が本文で果たされている | WARNING |
| 7 | 禁止事項 | ライターの禁止事項に違反していない | FAIL |

---

## 判定ルール

- **FAIL が1つでもある** → NG（差し戻し）
- **WARNING が3つ以上** → NG（差し戻し）
- **WARNING が2つ以下** → PASS

---

## PASS時の出力

```
✅ 記事チェック PASS

検査対象: article-writer
結果: PASS
WARNING: {あれば列挙、なければ「なし」}
```

---

## NG時の出力（1回目）

```
❌ 記事チェック NG

不合格項目:
  - #1 文字数: 1,800文字（基準: 3,000±10% = 2,700〜3,300文字）
  - #5 根拠なし数字: 「9割の人が挫折する」に出典なし

修正指示:
  - 文字数を3,000文字以上に増量すること
  - 「9割」の根拠を示すか、数字を使わない表現に書き換えること
```

---

## NG時の出力（2回連続）

```
🚨 エスカレーション: 記事チェック 2回連続NG

問題:
  - 文字数が依然として2,100文字（基準: 2,700〜3,300文字）

1回目の差し戻し内容:
  - 文字数を3,000文字以上に増量するよう指示

2回目の結果:
  - 300文字増えたのみ。各セクションの増量が不十分

推定原因:
  - テーマに対する情報収集が不足している可能性
```

---

## 注意事項

- 品質基準は厳格に適用する。「だいたいOK」は認めない
- チェックは客観的な基準に基づく。主観的な「良し悪し」で判定しない
- WARNING は記録しておき、傾向があればユーザーに報告する
```

---

## 重大度の設計：FAILとWARNINGを使い分ける

チェック項目には**重大度**を設定します。これが地味に重要です。

**FAIL** は「これがダメなら絶対に通さない」項目。文字数、事実の正確性、禁止事項違反など。1つでもFAILがあれば即NG。

**WARNING** は「望ましくないが、致命的ではない」項目。構成の改善余地、具体例の不足など。WARNINGが少数なら通すが、積み重なったらNG。

この使い分けがないと、些細な問題で毎回差し戻しが発生して業務が止まります。逆にすべてWARNINGにすると、重大な問題を見逃します。

判定ルールの例：
- FAIL 1つ以上 → NG
- WARNING 3つ以上 → NG
- WARNING 2つ以下 → PASS（WARNINGの内容は報告）

---

## 2回連続FAIL → エスカレーション

チェッカーの最も重要なルールが、**2回連続FAILでエスカレーション**です。

なぜ2回なのか。

1回目のFAILは「うっかりミス」かもしれません。修正指示を出せば直る可能性が高い。だから差し戻してリトライさせます。

でも、同じ問題が2回連続で直らないなら、**そのスキルの能力では解決できない問題**です。3回、4回とリトライさせても無駄にトークンを消費するだけ。

この時点で上位（部長スキル、またはユーザー本人）にエスカレーションします。

```
1回目NG → 具体的な修正指示付きで差し戻し
2回目NG → 上位にエスカレーション（問題・試行内容・推定原因を報告）
```

エスカレーションのフォーマットには必ず以下を含めます：

1. **何が問題か**（具体的なチェック項目と数値）
2. **1回目に何を指示したか**（どんな修正を求めたか）
3. **なぜ改善できなかったか**（推定原因）

これがないと、エスカレーションを受けた側が状況を把握できません。「ダメでした」だけでは困ります。人間の組織と同じで、報告には「事実・経緯・推定原因」の3点セットが必要です。

---

## チェッカーの配置：パイプラインのどこに入れるか

チェッカーは「工程の間」に挟みます。

```
記事ライター → [記事チェッカー] → 公開
                    ↓NG
               差し戻し（1回目）
                    ↓2回連続NG
               エスカレーション
```

もっと複雑なパイプラインでは、各ステップの後ろにチェッカーを入れます：

```
データ収集 → [Check 1] → 分析 → [Check 2] → 実行 → [Check 3]
```

こうすると、問題が早期に発見されます。データ収集が不完全なまま分析に進んでも、ゴミからは何も生まれません。**各工程の出口にゲートを置く**のが鉄則です。

---

## 複数スキルをチェックする統合チェッカー

スキルが増えてくると、スキルごとに個別のチェッカーを作るのは大変です。実用的なパターンとして、**1つのチェッカーで複数スキルの出力をチェックする**設計があります。

```markdown
## Check 1: ライターの出力チェック

| # | チェック項目 | 基準 | 重大度 |
|---|---|---|---|
| 1-1 | 文字数 | 指定範囲内 | FAIL |
| 1-2 | 構成 | H2が5個以上 | FAIL |
| 1-3 | 事実確認 | 数値に根拠あり | FAIL |

## Check 2: データ収集の出力チェック

| # | チェック項目 | 基準 | 重大度 |
|---|---|---|---|
| 2-1 | ファイル形式 | CSVヘッダーが正しい | FAIL |
| 2-2 | データ件数 | 最低10件以上 | FAIL |
| 2-3 | 数値の妥当性 | 範囲内の値 | WARNING |

## Check 3: 実行結果の出力チェック

| # | チェック項目 | 基準 | 重大度 |
|---|---|---|---|
| 3-1 | コミット確認 | 今日のコミットが存在する | FAIL |
| 3-2 | 変更件数 | 指示された件数と一致 | FAIL |
| 3-3 | 品質基準 | ファイルサイズが変更前以上 | FAIL |
```

チェック番号をセクション別に振ることで（1-1, 1-2... / 2-1, 2-2...）、どのスキルのどの項目が問題だったかを一意に特定できます。

---

## チェッカーが「手を抜く」問題への対処

ここで1つ問題があります。チェッカー自身もAIなので、**チェッカーが甘くチェックする**可能性があります。

これに対する実用的な対策は3つ。

### 1. チェック項目を「数値化」する

「品質が高い」→ 判定できない
「3,000文字以上」→ 数えれば判定できる

チェッカーに判断の余地を与えないことが最善策です。

### 2. 検証コマンドを書く

可能なら、チェック項目をシェルコマンドやスクリプトで検証させます。

```markdown
### 検証方法
文字数の確認:
  - 出力テキストの文字数を実際にカウントする
  - ライターの自己申告値と比較する
  - 差が10%以上あれば自己申告の信頼性にFLAG

コミットの確認:
  git log --oneline -1 で今日のコミットを確認
```

AIの自己申告ではなく、**実際のファイルやデータを読んで検証する**手順を書くと精度が上がります。

### 3. チェッカーの結果を人間がたまに抜き打ち確認する

最終的な品質保証は人間（社長）が担います。毎回見る必要はありませんが、月に1回くらいチェッカーのPASS判定を抜き打ちで確認すると、チェッカーの精度も把握できます。

---

## まとめ：チェッカーなしの運用は「目をつぶって走る」のと同じ

この章で伝えたかったことは1つだけです。

**スキルを作ったら、必ずチェッカーも作る。**

ライターだけ雇ってチェッカーなしで運用すると、品質は確実に崩壊します。最初は問題ないように見えても、10回、20回と繰り返すうちに「AIなりの手の抜き方」が蓄積していきます。

ライターとチェッカーはペアです。社員を1人雇ったら、その仕事を検査する人も同時に雇う。これがAI会社運用の基本中の基本です。

次章では、このライター＋チェッカーのペアを**パイプライン**として連結し、自動で流れる仕組みを作ります。


# 第4章：パイプラインを組む — 自動で流れる仕組み

ここまでで、あなたの手元には「社員スキル」と「チェッカースキル」ができているはずです。でもこの2つ、毎回手動で呼び出していませんか？

「データ取ってきて」→ 結果を確認 →「分析して」→ 結果を確認 →「実行して」→ 結果を確認……

これ、人間がやっている限り「自動化」とは言えません。

この章では、スキルを**連鎖的に自動実行**する「パイプライン」の作り方を解説します。一度セットアップすれば、シェルスクリプト1本で全工程が人間の介入なしに流れるようになります。

---

## パイプラインとは何か

パイプラインとは、**スキルの連鎖実行**です。

工場の生産ラインを想像してください。原料が入ってきて、加工されて、検品されて、出荷される。途中で人間が「次の工程に回して」と言わなくても、ベルトコンベアが自動で流してくれる。

AI会社のパイプラインも同じです。

```
データ収集 → [品質チェック] → 分析 → [品質チェック] → 実行 → [品質チェック]
```

各ステップの間にチェッカーが挟まっていて、品質が基準を満たしたら次のステップへ、ダメなら差し戻し。これが全自動で流れます。

ポイントは**「各ステップ後にチェックを入れる」**こと。チェックなしで流すと、Step 1でデータ取得に失敗したのにStep 2が空のデータで分析を始めて、Step 3が意味不明なアクションを実行する……という惨事が起きます。第3章の教訓を忘れないでください。

---

## 3ステップパイプラインの基本形

最もシンプルで実用的なパイプラインは、3ステップ構成です。

### Step 1: データ収集

外部からデータを取ってくる工程です。API呼び出し、スクレイピング、ファイル読み込みなど。

```markdown
# data-collector スキルの概要
- 入力: データソースの指定（API、CSV、外部ツール等）
- 処理: 指定されたソースからデータを取得し、所定のフォルダに保存
- 出力: CSVファイル群 + 収集ログ
```

### Check 1: データ検証

収集したデータが「使える品質」かを検査します。

```
検査項目:
- 必須ファイルが全て存在するか
- 各ファイルの行数が最低基準を満たすか（空データでないか）
- 収集ログにエラーが記録されていないか
```

ここがFAILしたら、Step 2には進みません。データがなければ分析もできないので、当然ですね。

### Step 2: 分析

収集したデータを読み込んで、レポートを生成する工程です。

```markdown
# analyzer スキルの概要
- 入力: Step 1で収集したデータ群
- 処理: データを多角的に分析し、優先度付きのアクションリストを生成
- 出力: 分析レポート（Markdown）+ Top 5 アクション
```

### Check 2: レポート検証

分析レポートが「実行可能な品質」かを検査します。

```
検査項目:
- 必要なセクションが全て含まれているか
- Top 5 アクションが具体的か（ページ名・変更内容・数値が明記されているか）
- アクションが抽象的すぎないか（「SEOを改善する」はNG、「/services/dmm/のtitleを〇〇に変更」はOK）
```

「改善しましょう」みたいなぼんやりした分析が出てきたら、ここで弾きます。

### Step 3: 実行

分析レポートのTop 5アクションを、実際にコードやコンテンツに反映する工程です。

```markdown
# executor スキルの概要
- 入力: 分析レポート（Top 5 アクション）
- 処理: 各アクションをファイル編集・git commit & push で実行
- 出力: 変更済みファイル群 + 実行ログ
```

### Check 3: 実行検証

変更が正しく反映されたかを検査します。

```
検査項目:
- 今日のgitコミットが存在するか
- Top 5 全件が実行されたか
- 変更内容が品質基準を満たすか（文字数、メタタグの形式等）
- git push 済みか（uncommitted changesが残っていないか）
```

## Agent toolでスキルを連鎖させる方法

Claude Codeには「Agent tool」という仕組みがあります。これは**サブエージェントを生成して、別のスキルを実行させる**機能です。

パイプラインスキルの中で、こんなふうに書きます：

```markdown
## Step 1: データ収集

Agent tool で以下のサブエージェントを生成する:

  description: "データ収集"
  prompt: |
    以下のSKILL.mdを読み、その指示に従ってデータ収集を実行せよ。
    SKILL.mdパス: ~/.claude/skills/data-collector/SKILL.md

    データ保存先: {共有フォルダのパス}
    完了したら「STEP1_DONE」と出力せよ。

サブエージェント完了後、Check 1 を自分で直接実行する。
```

重要なのは、**チェックはパイプラインスキル自身が行う**という点です。サブエージェントに「自分で品質チェックもして」と任せると、甘い自己採点になりがちです。第三者チェックの原則を思い出してください。

### サブエージェントに渡す情報

サブエージェントは「別の社員」です。パイプラインスキル（指揮者）から社員スキルに仕事を渡すとき、以下を明確に伝える必要があります：

1. **何をするか** — SKILL.mdのパスを渡して「この指示書に従え」
2. **データの場所** — 共有フォルダのパスを明示
3. **完了の合図** — 「STEP1_DONE」のようなキーワードを出力させる

逆に、渡す必要がないのは「なぜやるか」です。社員は手順通りに動けばいい。目的を理解して判断するのは部長の仕事です（次章で解説します）。

## シェルスクリプトで完全自動化する

ここまではClaude Codeの中でパイプラインを動かす方法でした。でも本当の自動化は**シェルスクリプト1本で、ターミナルから一発実行**です。

### 基本構造

```bash
#!/bin/bash
set -euo pipefail

TODAY=$(date +%Y-%m-%d)
DATA_DIR="/path/to/data/$TODAY"
SKILLS_DIR="$HOME/.claude/skills"
REPO_DIR="$HOME/projects/my-site"

echo "=== パイプライン開始: $TODAY ==="

# Step 1: データ収集
SESSION_ID=$(claude -p "
以下のSKILL.mdを読み、データ収集を実行せよ。
SKILL.mdパス: $SKILLS_DIR/data-collector/SKILL.md
データフォルダ: $DATA_DIR
完了したら「STEP1_DONE」と出力せよ。
" --allowedTools "Bash,Read,Write,Glob,Grep" \
  --output-format json 2>/dev/null | jq -r '.session_id')

# Check 1: データ検証
for f in data1.csv data2.csv log.txt; do
  if [ ! -f "$DATA_DIR/$f" ]; then
    echo "FAIL: $f が見つかりません"
    exit 1
  fi
done
echo "Check 1 PASS"

# Step 2: 分析（同じセッションを再利用）
claude -p "
以下のSKILL.mdを読み、分析レポートを作成せよ。
SKILL.mdパス: $SKILLS_DIR/analyzer/SKILL.md
データフォルダ: $DATA_DIR
" --resume "$SESSION_ID" \
  --allowedTools "Bash,Read,Write,Glob,Grep" \
  --output-format json 2>/dev/null > /dev/null

echo "Step 2 完了"

# ... Step 3, Check 3 も同様に続く
```

### ポイント解説

**`set -euo pipefail`** — エラーが出たら即座にスクリプトを停止します。データ収集に失敗したのに分析に突入する事故を防ぎます。

**`--allowedTools`** — サブエージェントに使わせるツールを制限します。データ収集にはAPI呼び出し用のツールを許可し、実行ステップにはファイル編集ツールを許可する、というように工程ごとに権限を絞れます。

**`--resume "$SESSION_ID"`** — 同じセッションを引き継ぐことで、Step 1で取得したデータのコンテキストをStep 2以降でも使えます。毎回ゼロから状況を説明し直す必要がなくなります。

**チェック処理はシェルスクリプト側で** — 「ファイルが存在するか」「行数が足りているか」程度の検査は、bashの `if` 文で十分です。Claude APIを呼ぶよりはるかに速く、確実です。

### cronで定期実行する

シェルスクリプトができたら、あとはcronに登録するだけです。

```bash
# 毎週月曜 9:00 にパイプラインを実行
0 9 * * 1 /path/to/run-pipeline.sh >> /path/to/pipeline.log 2>&1
```

これで毎週月曜の朝、あなたが何もしなくても、データ収集→分析→実行→品質チェックが全自動で回ります。

## 実例：SEOサイトの自動改善パイプライン

ここからは、実際に運用しているSEOサイトの週次改善パイプラインを紹介します。

### 背景

ある英語学習サイト（200ページ以上のコンテンツ）のSEOを自動改善するパイプラインです。Search ConsoleとGA4のデータをもとに、毎週自動でtitle改善・内部リンク追加・メタタグ修正を行っています。

### パイプライン全体像

```
[GAS: 毎朝6時] SC/GA4データ → Google Driveに保存
         ↓
[Step 1] Ahrefsデータ収集（MCP経由）
         ↓
[Check 1] 必須CSV6ファイルの存在・行数チェック
         ↓
[Step 2] SEO分析レポート生成（10セクション + Top 5アクション）
         ↓
[Check 2] セクション数・Top 5の具体性チェック
         ↓
[Step 3] Top 5アクション自動実行 → git commit & push
         ↓
[Check 3] コミット存在・実行完了・品質基準チェック
         ↓
[Step 4] 部長がサイト状況レポートを生成
```

### 各ステップの詳細

**Step 1: データ収集**

SC/GA4のデータはGoogle Apps Script（GAS）で毎朝自動取得しています。パイプラインが取るのはAhrefsのデータだけ。MCP（Model Context Protocol）でAhrefs APIに接続し、自サイトと競合のキーワード・バックリンク・トラフィックデータを取得します。

ここで重要なのは**データソースの分離**です。GASとパイプラインを完全に分けることで、「GASが動かなくてもAhrefsだけで分析を進める」という柔軟性が生まれます。

**Check 1: データ検証**

```bash
# 必須ファイルの存在チェック
for f in queries.csv pages.csv dates.csv ga4_traffic.csv ga4_pages.csv collection_log.txt; do
  if [ ! -f "$DATA_DIR/$f" ]; then
    echo "必須ファイル欠落: $f"
    exit 1
  fi
done
```

シンプルですが、これがないと空データで分析が始まります。「シンプルなチェックほど大事」という原則です。

**Step 2: SEO分析**

analyzerスキルが10セクションのレポートを生成します：

- キーワード順位の変動
- 高インプレ・低CTRページ（改善余地が大きい）
- GA4のエンゲージメントデータ
- 競合との比較
- バックリンクプロファイル

そして最も重要な**Top 5アクション**。ここでは「収益ティア」という概念でスコアリングしています。

```
Tier 1（収益直結ページ）: 重み 3.0
Tier 2（収益支援ページ）: 重み 1.5
Tier 3（SEOサポート）:    重み 1.0
```

同じ改善余地でも、収益に直結するページの施策が優先されます。「SEOスコアを上げる」ではなく「収入を増やす」が目的だからです。

**Step 3: 実行**

Top 5アクションを実際にHTMLファイルに反映します。titleの書き換え、meta descriptionの改善、内部リンクの追加、JSON-LD構造化データの修正など。変更後は自動でgit commit & pushし、GitHub Pagesに反映されます。

**Step 4: 部長レポート**

全ステップ完了後、部長スキルがサイト状況のサマリーを生成します。日別トレンド、GA4トラフィック、人気ページ、検索クエリの状況を俯瞰できるレポートです。これが次回パイプラインの判断材料にもなります。

### このパイプラインの成果

- **週次実行**: 毎週5つのSEO改善が自動で実行される
- **判断の一貫性**: 収益ティアに基づくスコアリングで、感覚ではなくデータで優先度を決定
- **品質保証**: 3段階のチェックで、不完全な改善がデプロイされることを防止
- **人間の作業**: ゼロ。シェルスクリプト1本を実行するだけ（cronなら実行すら不要）

## エラーハンドリングと中断ルール

パイプラインは「途中で止まる」ことを前提に設計してください。止まらないパイプラインは、壊れたデータを最後まで流してしまうパイプラインです。

### 中断ルール

```
Check FAIL → 1回リトライ → 再FAIL → パイプライン中断 → 部長にエスカレーション
```

2回連続FAILしたら人間（部長）に判断を委ねます。3回、4回とリトライしても同じ結果になることがほとんどだからです。

### エラーの種類と対処

| エラー種別 | 対処 |
|---|---|
| データソース一時障害 | 1回リトライ。ダメなら利用可能なデータだけで続行 |
| 分析の品質不足 | 不足セクションを指定して再生成 |
| 実行時のgitエラー | 状態確認 → `git pull --rebase` → 再push |
| 全データ取得不能 | パイプライン中断 → 部長（次章参照） |

### パイプラインの「部分実行」

全部止めるか全部通すかの二択ではありません。「Step 1のデータAは取れなかったけどデータBは取れた」場合、データBだけで分析を続行する判断もあり得ます。

ただし、その判断は社員スキルにはできません。「データが足りないけど続行する」という**例外的な判断**は、次章で解説する「部長」の仕事です。

---

## まとめ

- パイプラインは**スキルの連鎖実行**。工場のベルトコンベア
- 基本形は**データ収集→分析→実行**の3ステップ。各ステップ間にチェッカー
- Agent toolでサブエージェントを生成し、SKILL.mdのパスを渡す
- シェルスクリプト化すれば、ターミナル1コマンドで全自動実行
- cronに登録すれば、人間が何もしなくても毎週回る
- エラー時は**「止める」が正解**。壊れたデータを流し続けるより100倍マシ

次の章では、パイプラインが止まったときに**判断を下す存在**——「部長」の作り方を解説します。


# 第5章：部長を雇う — 判断できるAIを作る

前章でパイプラインを作りました。データ収集→分析→実行が自動で流れる。チェッカーが品質を検査する。素晴らしい。

でも、こんな場面が必ず来ます。

「データソースAが落ちている。データBだけで続行するか、全部止めるか？」

チェッカーは「FAILです」と言うだけ。社員スキルは手順通りに動くだけ。**誰も判断してくれない。**

結局あなた（社長）がターミナルに戻ってきて、状況を見て、指示を出すことになる。これでは「自動化」ではなく「半自動化」です。

この章では、**状況を判断して自分で対処できるAI——「部長」**の作り方を解説します。

---

## 社員と部長の決定的な違い

AI会社の3層構造を思い出してください。

```
社長（人間）
  ↓ 例外中の例外のみ介入
部長スキル（判断者）
  ↓ 社員に指示、問題を対処
社員スキル（実行者）
  ↓ 手順通りに実行
```

社員と部長の違いは、一言でいうと**「手順か判断か」**です。

| | 社員スキル | 部長スキル |
|---|---|---|
| 動き方 | 与えられた手順を忠実に実行 | 状況を見て何をすべきか決める |
| 失敗時 | 1回リトライ → 上に報告 | 根本原因を推定し、代替手段を採用 |
| 権限 | 自分の担当工程のみ | 配下の社員全員を指揮 |
| 知識 | SKILL.md内の手順 | 部門の目標・KPI・対処パターン |
| 呼ばれ方 | パイプラインから自動で | エスカレーション時 or コマンドで |

具体例で考えましょう。SEOパイプラインでGA4のデータが取れなかった場合：

**社員（チェッカー）の反応:**
> Check 1 FAIL — ga4_traffic.csv が存在しません。リトライします。
> ……再FAIL。エスカレーションします。

**部長の反応:**
> GA4データは取れていないが、AhrefsデータとSearch Consoleデータは取れている。
> GA4なしでもキーワード分析とバックリンク分析は実行可能。
> → analyzerに「GA4なし」を伝えて、利用可能なデータだけで分析を続行する。
> → 完了後レポートに「GA4データ未取得のため一部分析はスキップ」と警告を記載する。

この違いわかりますか？ 社員は「取れない→ダメ」で止まる。部長は「取れないけど、ここまではできる」と判断して進める。

---

## 部長スキルの設計パターン

部長スキルのSKILL.mdは、社員スキルとは構造が大きく異なります。

### 社員スキルの構造（おさらい）

```markdown
# 記事ライター

## 入力
- テーマ、ターゲット読者、文字数

## 手順
1. テーマをリサーチする
2. アウトラインを作る
3. 本文を書く
4. 校正する

## 出力
- Markdown記事
```

手順が明確。入力→処理→出力が一直線。

### 部長スキルの構造

```markdown
# SEO部門 部長AI

## 部門の目標
アフィリエイト収入の増加。検索順位改善はその手段。

## 役割
社員スキルは「決められた手順を実行する」のみ。
部長は「状況を判断して何をすべきか決める」。

## 管轄する社員
- data-collector（データ収集）
- analyzer（分析）
- executor（実行）
- checker（品質検査）

## 対処パターン一覧
### パターン1: データ収集の一部失敗
### パターン2: 分析の品質不足
### パターン3: 実行時のエラー
### パターン4: 同じ問題の繰り返し（慢性障害）
### パターン5: 部長でも解決できない場合

## KPI管理
## 定期レビュー
## コマンド一覧
```

手順ではなく**判断基準**が書かれています。「こういう状況ならこう判断する」のパターン集です。

---

## 部長に持たせる4つの権限

部長スキルには、社員にはない権限を持たせます。

### 1. 社員スキルの呼び出し（Agent tool）

部長は配下の社員を自由に呼び出せます。

```markdown
## 管轄する社員

| 社員 | 役割 |
|---|---|
| writer | 投稿ライター |
| schedule-post | 予約投稿 |
| data-collector | データ収集 |
| analyzer | 分析・パターン発見 |
| checker | 品質検査 |
```

エスカレーションを受けたとき、部長は原因を分析して、該当する社員に再実行の指示を出します。このとき、社員には「こういう条件で、ここだけ再実行しろ」と具体的に指示できます。

### 2. KPI確認と方針決定

部長は数字を見て方針を決めます。

```markdown
## KPIの構造

【主KPI】月間アフィリエイト収入（円）
  ↑ 社長が月1回確認して部長に伝える

【先行指標KPI】毎週自動計測
  KPI-1: 収益直結ページの月間クリック数
  KPI-2: 収益直結ページの平均CTR
  KPI-3: ランキングページの平均掲載順位
```

「主KPI（売上）を上げるために、先行指標（クリック数・CTR）を改善する」という因果関係を部長が理解しているので、「CTRが低いページのtitle改善を優先しろ」という判断ができます。

### 3. 失敗時のリカバリー判断

これが部長の最大の価値です。

```markdown
## 対処パターン

### パターン1: データ収集の一部失敗

判断フロー:
  データBは取得できているか？
    ├── YES → データBのみで分析を続行
    │         （analyzerに「データAなし」を伝えて実行）
    └── NO  → 全データ取得失敗 → 社長へ報告
```

**判断フローを明文化**するのがコツです。「状況を見て適切に判断して」と曖昧に書くと、AIは無限に迷うか、的外れな判断をします。

「こういう状況なら→こうする」というif-then形式で、想定される障害パターンを全て書き出してください。最初から完璧でなくていい。運用しながら「こういうケースもあったのか」とパターンを追加していけばOKです。

### 4. 社長への報告

部長でも解決できない場合にだけ、社長（人間）に報告します。

```markdown
## 社長への報告フォーマット

🚨 部長報告: {問題の要約}

状況:
  {何が起きているか}

試みた対処:
  1. {対処1} → {結果}
  2. {対処2} → {結果}

根本原因（推定）:
  {なぜ解決できないか}

お願いしたい対応:
  {具体的な手順。1〜3ステップ以内で簡潔に}
```

ここで重要なのは **「お願いしたい対応」を具体的に書かせる** ことです。「問題が起きました、どうしましょう？」という丸投げ報告は却下。「APIキーを再発行して、以下のコマンドで設定してください」と、社長が5分で対応できる形にまとめさせます。

---

## コマンド体系の設計

部長スキルには**コマンド**を持たせます。社長が部長に指示を出すときのインターフェースです。

### 基本コマンド

```
/bucho post    → 投稿バッチの作成を指揮
/bucho review  → 週次レビューの実行
/bucho kpi     → KPIの確認・目標設定
/bucho learn   → 知識ベースの更新
```

各コマンドの実行手順をSKILL.md内に詳細に記述します。

### `/bucho post` — 投稿バッチ作成

部長が最も頻繁に使うコマンドです。SNS運用の部長を例に見てみましょう。

```markdown
### `/bucho post` — 7日分バッチ一括作成

#### 5段階パイプライン

① 部長（方針決定・テーマ配分）
  → ② writer（投稿文作成）
    → ③ checker（品質チェック）
      → ④ schedule-post（予約投稿）
        → ⑤ 部長（ログ記録・バッチ保存）
```

注目してほしいのは、パイプラインの**最初と最後に部長がいる**こと。方針を決めるのも部長、結果を記録するのも部長。社員は中間工程を担当するだけです。

### `/bucho review` — 週次レビュー

```markdown
### `/bucho review` — 週次レビュー

#### 実行手順

Step 1: data-collector にデータ取得を指示
Step 2: checker でデータ品質を検査
Step 3: analyzer に分析・パターン発見を指示
Step 4: 部長が分析結果を確認し、判断を加える
Step 5: 知識ベースを更新
Step 6: レポートを出力
Step 7: git commit & push
```

レビューは「データを見る」だけではありません。**知識ベースを更新する**（第6章で詳述）ことで、次の投稿作成が前回より賢くなります。これが自己改善ループの核心です。

### `/bucho kpi` — KPI確認

```markdown
### `/bucho kpi`

#### 実行手順

1. 最新データから現在値を計測
2. 前月実績と比較
3. 目標達成率を計算
4. 翌月目標を自動設定:
   - 達成率100%以上 → 翌月 +20%
   - 達成率80〜99% → 目標据え置き
   - 達成率80%未満 → 翌月 +10%に下方修正
5. 達成に必要なアクションを部長の判断で提示
```

目標設定を**ルール化**するのがポイントです。「順調なら攻め、停滞なら現実路線」という方針を数値で定義しておけば、部長が感覚で判断することなく、一貫した目標管理ができます。

## 実例：SNS部門部長の1週間

SNSアカウント運用を統括する部長AIの動きを、1週間で追ってみましょう。

### 月曜日：週次レビュー

部長は先週の投稿パフォーマンスデータを`data-collector`に取得させます。

```
■ 週次レビュー（2026-03-17）

パフォーマンスサマリー:
  投稿数: 14
  平均インプレッション: 5,200
  平均エンゲージメント率: 4.8%
  前週比: +12%

ベスト投稿:
  「TOEICで900点取った人の○○...」 → imp 12,000 / eng 6.2%（Sランク）
  勝因: 数字フック + 常識否定の組み合わせ

仮説検証:
  H3「疑問形フックはエンゲージメント率が高い」
  → サンプル8/10。中間結果: +18%。あと2投稿でテスト完了予定。

知識ベース更新:
  - patterns.md: 「数字+常識否定」の組み合わせを勝ちパターンに追加
  - hooks.md: 疑問形フックのステータスを「テスト中」に更新
```

analyzer が見つけたパターンを部長が**判断して**知識ベースに登録しています。「本当にこのパターンは再現可能か？ サンプル数は十分か？」という妥当性チェックが部長の仕事です。

### 水曜日：投稿バッチ作成

部長が `/bucho post` で来週分14投稿の作成を指揮します。

```
Step 1: 方針決定
  - テーマ配分: 7テーマを2投稿ずつ（前週と重複しないよう調整）
  - 仮説テスト: 14投稿中4投稿を実験枠に割り当て
    - H3（疑問形フック）のテスト残り2投稿を含める
  - Factカテゴリ: 研究系40% / 実践系30% / 比較系20% / リスト系10%

Step 2: writer に14投稿の作成を指示
  - 各投稿にテーマ・使用するFact・テンプレートを指定
  - 実験枠には仮説タグ（H3）を付与

Step 3: checker が全14投稿を品質検査
  - 2投稿がNG（文字数超過、Factの引用なし）→ writer に差し戻し
  - 修正後、全14投稿PASS

Step 4: ユーザー承認後、schedule-post で予約投稿
```

ここでのポイントは、部長が**テーマ配分と実験配分を判断している**こと。「安定投稿70%・実験投稿30%」というバランスを取り、仮説検証のためのA/Bテスト設計まで部長がやっています。社員には「この投稿を書け」と指示するだけです。

### 金曜日：中間パフォーマンス確認

```
■ 中間チェック（2026-03-19）

月曜投稿:
  朝: 「○○を知らない人が多い...」 → imp 4,800 / eng 3.9%（B）
  夕: 「○○は実は△△だった...」 → imp 7,200 / eng 5.1%（A）
  傾向: 夕方投稿の方がエンゲージメント率が高い

火曜投稿:
  朝 [H3テスト]: 「なぜ○○なのか？」 → imp 6,100 / eng 5.8%（A）
  → H3仮説の中間データ更新: サンプル9/10、平均+22%

アクション: 特になし。来週月曜のレビューで最終判定。
```

部長は**「アクションが必要か不要か」も判断します**。中間チェックで異常値がなければ「何もしない」が正解。部長が毎回何かを変えようとすると、ノイズが増えるだけです。

### この1週間で起きたこと

- 過去のデータから勝ちパターンを**発見**
- 仮説検証を**進行**
- 来週の投稿方針を**決定**
- 社員に具体的な指示を**配信**
- 知識ベースを**更新**

社長（人間）がやったことは？ **水曜日に投稿内容を承認しただけ**です。

## 部長スキル設計のチェックリスト

部長スキルを作るとき、以下の項目が含まれているか確認してください。

- [ ] **部門の目標**が明記されているか（数字で）
- [ ] **管轄する社員の一覧**があるか
- [ ] **対処パターン**が最低3つ以上あるか
- [ ] 各パターンに**判断フロー（if-then）**が書かれているか
- [ ] **社長への報告フォーマット**があるか
- [ ] 報告には「お願いしたい対応」が含まれるか（丸投げ禁止）
- [ ] **KPIの定義**があるか
- [ ] KPIの**目標設定ルール**が数値化されているか
- [ ] **コマンド一覧**（post / review / kpi等）があるか
- [ ] 各コマンドの**実行手順**が明記されているか

## 部長の暴走を防ぐ

部長スキルに大きな権限を持たせると、当然リスクも生まれます。

### 権限と制約のテーブル

```markdown
| できること | できないこと |
|---|---|
| 配下社員の呼び出し・再指示 | 他部門への干渉 |
| 分析・実行の判断 | ユーザーのアカウント設定変更 |
| 代替手段での続行判断 | API設定・権限の変更 |
| リポジトリのファイル変更 | 社長への無断アクション |
| スキルファイルの改善提案 | — |
```

「できること」と「できないこと」を**明示的に書く**のが重要です。特に「社長への無断アクション」の禁止は絶対です。部長が「よかれと思って」勝手にデプロイしたり、外部サービスに何かを送信したりするのを防ぎます。

### エスカレーションの強制

部長がどれだけ賢くても、**解決できない問題は必ず存在します**。そのとき黙って止まるのではなく、社長に報告させる仕組みが必要です。

```markdown
## 社長へ報告すべき場合（必ず守ること）

- 全データソースが取得不能
- gitリポジトリが破損・コンフリクト解消不能
- スキル自体の根本的な設計ミス
- 2回のリカバリー試行でも解決しない問題
```

「報告すべき場合」を列挙しておけば、部長が「自分で解決しなきゃ」と延々リトライし続ける事態を防げます。

---

## まとめ

- **社員は手順を実行する。部長は状況を判断する。** この違いが組織を自律的にする
- 部長に持たせる権限は4つ：**社員の指揮、KPI管理、リカバリー判断、社長への報告**
- 対処パターンは**if-then形式で明文化**する。「適切に判断して」はNG
- コマンド体系（`/bucho post` `/bucho review` `/bucho kpi`）で社長とのインターフェースを定義
- 部長の暴走防止には**権限テーブル**と**強制エスカレーション条件**を設定

次の章では、部長が判断の根拠にする**「知識ベース」**の育て方を解説します。部長が賢くなり続ける仕組みです。


# 第6章：知識ベースを育てる — 自己改善するAI組織

## スキルだけでは「毎回ゼロから」になる

ここまでで、社員スキル、チェッカー、パイプライン、部長と、AI会社の組織構造を一通り作ってきました。でも、このままだと1つ大きな問題があります。

**AIは毎回ゼロからスタートする。**

人間の組織だったら、「前回あの施策うまくいったよね」「あのやり方はダメだったから変えよう」という暗黙知が自然に蓄積されていきます。でもAIにはそれがない。スキルに書いてあること以外は、毎回まっさらな状態で仕事を始めます。

たとえばSNS運用を考えてみてください。投稿を100回作っても、「どんな書き出しがウケるのか」「どのテーマが伸びるのか」という知見がどこにも残らなければ、101回目も1回目と同じクオリティです。

これを解決するのが「知識ベース」です。

---

## 知識ベースの全体像

知識ベースは、ある部門が業務を通じて学んだことを構造的に蓄積する場所です。スキル（SKILL.md）が「やり方」を定義するのに対して、知識ベースは「何がうまくいくのか」を記録します。

ディレクトリ構成はこうなります：

```
{部門名}-knowledge/
├── INDEX.md              # ナビゲーション（どのファイルを読むべきか）
├── craft/                # テンプレート、フック、文体ルール
│   ├── templates.md      # 投稿テンプレート（使用回数・成績つき）
│   ├── hooks.md          # 書き出しパターン集
│   └── rules.md          # 文体・構造のルール
├── facts/                # 根拠あるデータ・統計
│   ├── INDEX.md          # Factストック一覧
│   └── {テーマ}.md       # テーマ別のFactファイル
├── hypotheses/           # テスト中の仮説
│   └── active.md         # 現在テスト中の仮説リスト
├── learnings/            # 確認済みパターン＋否定された信念
│   ├── patterns.md       # データで効果が確認されたパターン
│   └── false-beliefs.md  # 「効くと思ったが効かなかった」記録
├── posts/                # 過去の出力ログ＋パフォーマンス
│   ├── {account}-log.csv # 投稿ログとパフォーマンスデータ
│   └── batch-XXX.md      # 作成済みバッチ
└── themes/
    └── rotation.md       # テーマローテーション＋実績追跡
```

それぞれの役割を見ていきましょう。

---

## craft/ — 「技」を磨く

`craft/` は、コンテンツを作るための技術を蓄積するフォルダです。

### templates.md（テンプレート集）

投稿や記事のテンプレートを、**使用回数と成績つきで**管理します。ここがポイントです。テンプレートは「固定」ではなく「進化」するもの。

```markdown
### T1: 常識否定 → 解決策提示

[常識否定のフック（1行目）]

[なぜそれが間違いか（2-3行）]

[正しいアプローチ（2-3行）]

- 使用回数: 12 / 平均エンゲ率: 2.8%
```

使用回数とエンゲージメント率がテンプレートに紐づいているので、「どのテンプレートが実際に効くのか」がデータで分かります。成績が悪いテンプレートは改善するか廃止する。良いテンプレートは頻度を上げる。

### hooks.md（フックパターン集）

SNS投稿なら1行目、ブログ記事ならタイトル。最初の一文で読者の手を止められるかどうかが全てです。

フックパターンには3つのステータスがあります：

- **検証済み** — データで効果が確認された
- **テスト中** — まだサンプルが少ない
- **効果なし** — データで否定された

実際の知識ベースから例を出すと、こんな感じです：

```markdown
### 挑発+断定型（検証済み / imp実績: 3,500〜7,400）
「〇〇してる人、今すぐやめてください」
「〇割は〇〇を間違えてる」
- 読者の行動・信念を直接否定してスクロールを止める
- 複数投稿で3,500〜7,400 imp達成

### 慰め・優しい型（効果なし / imp実績: 600〜900）
「〜あなたのせいじゃないです」
- フックに「慰め」「共感」を置くと緊急性がゼロになる
- 本文内での共感はOKだが、1行目では使わない
```

「優しい書き出しは効くだろう」と思っていたのに、データを見たら全然ダメだった。こういう発見が知識ベースの価値です。

---

## facts/ — 投稿のネタ帳

`facts/` は、投稿や記事に使える「根拠のあるデータ」をテーマ別にストックしておく場所です。

たとえばSNSで英語学習について発信するなら：

```
facts/
├── INDEX.md          # 全テーマ一覧
├── listening.md      # リスニングに関するFact
├── vocabulary.md     # 語彙に関するFact
├── pronunciation.md  # 発音に関するFact
├── speaking.md       # スピーキングに関するFact
└── study-habits.md   # 学習習慣に関するFact
```

各Factには出典を必ずつけます。AIが「それっぽいこと」を言うのは簡単ですが、出典のない情報は嘘かもしれない。ファクトチェック済みのネタだけをストックしておくことで、投稿の信頼性を担保します。

ライタースキルが投稿を作るとき、このfactsフォルダからネタを引っ張ってきて、craftのテンプレートに当てはめる。この流れが自動化の基盤になります。

---

## hypotheses/ — 仮説を立てて検証する

ここがAI会社の「自己改善」の核心部分です。

`hypotheses/active.md` には、現在テスト中の仮説がフォーマットに沿って記録されています：

```markdown
### H1: 数字入りフックは数字なしより高インプレッション
- 仮説: フックに具体的な数字を含む投稿は、数字なしの投稿より
  インプレッションが30%以上高い
- 成功指標: 数字ありの平均インプレ / 数字なしの平均インプレ >= 1.3
- テスト方法: 同テーマで数字あり/なしを交互に投稿して比較
- 投稿数: 0/10（目標サンプル数）
- ステータス: テスト中
```

仮説には必ず「成功指標」と「テスト方法」を定義します。「なんとなく良さそう」ではなく、「何がどうなれば成功か」を事前に決めておく。これがないと、後からデータを都合よく解釈してしまいます。

### 仮説の一生

仮説のライフサイクルはこうなります：

```
気づき・アイデア
    ↓
active.md に仮説として登録（テスト中）
    ↓
投稿バッチに実験投稿を組み込む
    ↓
データが溜まったら検証
    ↓
┌─ 成功指標を満たした → patterns.md に昇格
└─ 満たさなかった → false-beliefs.md に記録
```

重要なのは「否定された仮説も記録する」ことです。

---

## learnings/ — 組織の記憶

### patterns.md（勝ちパターン）

仮説がデータで確認されると、ここに「勝ちパターン」として昇格します：

```markdown
### P1: 挑発+断定型フックは安定してimp 3,500以上
- 発見日: 2026-03-15
- 元仮説: H5
- エビデンス: 6投稿の平均 imp 4,800（対照群 1,200）
- 適用方法: バッチ作成時、14投稿中10投稿はこの型を使う
- 注意点: 同じ表現の使い回しは飽きられるのでバリエーションを出す
```

パターンには必ず「適用方法」を書きます。発見しただけでは意味がない。次にどう使うかまで書いて初めて、組織の知識として機能します。

### false-beliefs.md（否定された信念）

「効くと思ったけど効かなかった」を記録する場所です。これが地味に大事。

```markdown
### FB1: 慰め型フックはエンゲージメントが高い
- 当初の予測: 「あなたのせいじゃない」的な書き出しは共感を呼ぶ
- 実際の結果: imp 600〜900止まり。通常投稿の1/5以下
- 教訓: フックで「慰め」を置くと緊急性がゼロになりスルーされる
```

人間の組織でも「前やったけどダメだったよ」という情報は非常に価値がある。AIにもこの記憶を持たせることで、同じ失敗を繰り返さなくなります。

---

## 自己改善ループ — 知識ベースが回す好循環

知識ベースの真の力は、各フォルダが連携して「自己改善ループ」を形成することにあります：

```
実行（投稿する）
  ↓
データ収集（パフォーマンスを取得）
  ↓
分析（何が効いたか調べる）
  ↓
パターン発見（仮説の検証・新しい気づき）
  ↓
知識更新（patterns/false-beliefs/craft を更新）
  ↓
次の実行に反映（更新された知識で投稿を作る）
  ↓
（最初に戻る）
```

これを週次レビューとして部長スキルに実行させます。第5章で作った部長に「週次レビュー」コマンドを持たせて、このループを回すわけです。

### 週次レビューの流れ

1. `posts/` の投稿ログからパフォーマンスデータを読む
2. `hypotheses/active.md` のテスト中仮説を検証する
3. 結果に応じて `patterns.md` か `false-beliefs.md` を更新する
4. `craft/hooks.md` のステータスを更新する（検証済み / 効果なし）
5. `themes/rotation.md` のテーマ別成績を更新する
6. 新しい気づきがあれば新しい仮説を `active.md` に追加する

これが毎週自動で回ると、知識ベースは週を重ねるごとに賢くなっていきます。1ヶ月後には「どんなフックが効くか」「どのテーマが伸びるか」「どの時間帯がベストか」が全部データで分かっている状態になる。

---

## INDEX.md — Progressive Disclosure

知識ベースのファイルが増えてくると、「全部読む」のは非効率です。AIのコンテキストウィンドウも有限ですし、毎回全ファイルを読み込む必要はない。

そこで `INDEX.md` を「ルーター」として使います：

```markdown
# Knowledge Base ルーター

| やりたいこと | 読むファイル |
|---|---|
| 投稿を作成する | craft/templates.md → craft/hooks.md → facts/{テーマ}.md |
| 投稿を分析する | posts/log.csv → learnings/patterns.md → hypotheses/active.md |
| 仮説を検証する | hypotheses/active.md → posts/log.csv → learnings/patterns.md |
| 週次レビューする | 全ファイルを順に確認 |
```

タスクに応じて必要なファイルだけを読み込む。これを **Progressive Disclosure**（段階的開示）と呼んでいます。必要な情報に最短でたどり着ける設計です。

---

## テーマローテーション — 偏りを防ぐ仕組み

SNS運用で意外と難しいのが「テーマの偏り」です。ウケるテーマばかり投稿していると、フォロワーが飽きる。かといって全テーマ均等に回すと、伸びないテーマに時間を使いすぎる。

`themes/rotation.md` でこのバランスを取ります：

```markdown
| テーマ | 使用回数 | 平均エンゲ率 | 最終使用日 |
|---|---|---|---|
| リスニング上達法 | 8 | 3.2% | 2026-03-18 |
| 発音・フォニックス | 5 | 2.1% | 2026-03-15 |
| 語彙の増やし方 | 3 | 2.8% | 2026-03-10 |
```

ルールもシンプルです：

- 使用回数が少ないテーマを優先
- ただしエンゲージメント率が高いテーマは頻度を上げてもOK
- 同じテーマの連続投稿は避ける
- 1バッチ（7日分）内で7〜10テーマをカバーする

## メモリシステムとの違い

Claude Codeには「メモリ」（MEMORY.md）という仕組みがあります。知識ベースとメモリは似て非なるものです。

| | メモリ（MEMORY.md） | 知識ベース |
|---|---|---|
| **主体** | 社長（人間）の好みや方針 | 組織が業務で学んだ知識 |
| **更新者** | 社長が指示 or 自動保存 | 部長が週次レビューで更新 |
| **内容** | 「この書き方が好き」「この方針で」 | 「このパターンはimp 4,800」 |
| **永続性** | 長期間変わらない | データで常に更新される |
| **スコープ** | 全プロジェクト横断 | 特定部門に閉じる |

メモリは「社長の好み」、知識ベースは「組織の学び」。この2つを分けることで、AIは社長の意向を守りつつ、データに基づいた改善を自律的に続けられます。

---

## A/Bテスト配分 — 安定と実験のバランス

知識ベースを育てるには「実験」が必要です。でも実験ばかりしていたら本業のパフォーマンスが落ちる。

僕が採用しているのは **70:30ルール** です：

- **70%（14投稿中10投稿）**: 確認済みパターンを使った「安定投稿」
- **30%（14投稿中4投稿）**: 仮説をテストする「実験投稿」

安定投稿で確実にパフォーマンスを出しながら、実験投稿で新しいパターンを探る。実験が成功すれば安定投稿のレパートリーが増えるし、失敗しても全体への影響は30%に抑えられる。

---

## 知識ベースを始めるステップ

1. **ディレクトリを作る**: `{部門名}-knowledge/` を作成し、上記の構成でフォルダとファイルを作る
2. **INDEX.md を書く**: タスク別に「何を読むか」のルーターを作る
3. **craft/ を整備する**: 今使っているテンプレートやルールを書き出す
4. **facts/ にネタをストックする**: 最低10個のFactを出典付きで登録する
5. **hypotheses/ に仮説を3つ立てる**: 「〇〇すれば〇〇になるはず」を具体的に
6. **部長の週次レビューに組み込む**: 第5章の部長スキルにレビュー手順を追加する
7. **1ヶ月回す**: データが溜まれば、自然と patterns.md と false-beliefs.md が育っていく

最初は空っぽでも構いません。大事なのは「箱を用意しておく」ことです。箱があれば、運用するうちに中身が埋まっていく。箱がなければ、どんなに良い気づきも流れていってしまう。

知識ベースは、AI会社を「毎回ゼロからの作業」から「蓄積と改善の好循環」に変える仕組みです。次の第7章では、この知識ベースを含む全社の状態を一元管理する「COO」の作り方を見ていきます。


# 第7章：全社を統括する — COOと経営ダッシュボード

## 朝起きたらレポートが届いていた

ある朝、スマホを開いたらTelegramに通知が来ていました。

```
おはようございます。本日の状況です。

■ X部門
  投稿ストック: 残8件（3/24まで）→ 問題なし
  GitHub Actions: 全件成功
  昨日のパフォーマンス:
    朝投稿: imp 3,500 / eng 2.1%
    夕投稿: imp 2,800 / eng 1.8%

■ SEO部門
  本日のリライト対象: /services/cambly/
  title/meta改善: 10件完了
  前日比: クリック +12、インプレ +340

本日のアクション: 全部門通常運用。異常なし。
```

これを送ってきたのは、僕が雇ったCOO（最高執行責任者）です。もちろんAIの。

COOは毎朝、全部門の状況を確認して、部長たちにGoサインを出し、その結果を僕に報告してくれます。僕がやることは「おはよう」と声をかけることだけ。あとはCOOが全部回してくれます。

---

## COOの役割 — 部長の上に立つ存在

第5章で作った「部長」は、自分の部門を運用する責任者でした。X部門ならX部門、SEO部門ならSEO部門。でも部長には「部門間の調整」ができません。

- 「今月はSEOのほうが優先度高いから、X部門のAPI予算を少し回そう」
- 「2部門とも同時にKPI未達だ。どちらから手をつけるべきか」
- 「全社のコストが予算の80%に達している。ペースを落とすべきか」

こういう判断ができるのがCOOです。

組織の階層を整理するとこうなります：

```
社長（人間・あなた）
  └── COO（全社統括）
        ├── X部門 部長
        │     ├── ライタースキル
        │     ├── チェッカースキル
        │     └── データ収集スキル
        └── SEO部門 部長
              ├── ライタースキル
              ├── チェッカースキル
              └── データ収集スキル
```

社長が直接やり取りするのはCOOだけ。COOが部長に指示を出し、部長が社員スキルを使って実行する。

---

## COOスキルの設計

COOのSKILL.mdは、これまでのスキルより一段複雑です。でも構造は同じ。「役割」「管轄」「コマンド」「エスカレーション基準」を定義します。

```markdown
---
name: coo
description: |
  AI会社の最高執行責任者。全部門横断の経営管理を担当。
  KPI集約、予算・コスト追跡、部門間優先度調整、
  社長への統合レポート作成を行う。
---

# COO（最高執行責任者）

## 役割
社長の右腕として、全部門を横断的に管理する。
部長は自部門の運用に責任を持ち、COOは部門間の調整と全社最適化に責任を持つ。

**判断の基本原則**: 収益に近い施策を優先する。迷ったら社長に聞く。

## 管轄部門
| 部門 | 部長スキル | 主な役割 |
|------|-----------|---------|
| SEO部門 | seo-bucho | Webサイトの検索最適化・コンテンツ改善 |
| X部門 | x-bucho | SNSアカウントの運用・成長 |
```

ポイントは **「判断の基本原則」** です。COOは判断する権限を持つので、判断基準を明文化しておかないとAIが暴走します。「収益に近い施策を優先」「迷ったら社長に聞く」——このシンプルな基準が、COOの全ての判断の土台になります。

---

## 日次オペレーション — 「おはよう」で始まる1日

COOの最も重要な仕事は日次オペレーションです。毎朝の流れはこうなります：

### Step 1: 状況確認

COOがまず全部門の現在地を確認します：

- **投稿ストック残数**: schedule.jsonの未投稿件数を数える
- **GitHub Actions**: 自動投稿ワークフローの直近の成功/失敗を確認
- **前日のパフォーマンス**: 投稿のimp、エンゲージメント率を取得

### Step 2: 判断

状況に応じてCOOが判断を下します。ここがただのスクリプトとの違いです：

| 状況 | COOの判断 |
|------|----------|
| ストック残 <= 6件 | 部長にバッチ補充を指示 |
| 3日以上データ未収集 | データ収集スキルを実行 |
| 投稿データが10件以上 | 部長に週次レビューを指示 |
| GitHub Actions 失敗 | ログ確認 → 原因特定 → 修正 |
| 全て正常 | 日次タスクを部長に指示 |

### Step 3: 実行

判断に基づいて、部長スキルを呼び出します。COOは自分では「作業」をしません。部長に指示を出すだけ。部長がさらに社員スキルを使って実行する。

### Step 4: 報告

全部門の結果を統合して、社長に報告します。

```
■ X部門
  投稿ストック: 残4件 → バッチ補充しました（3/30まで延長）
  GitHub Actions: 全件成功

■ SEO部門
  本日のリライト: /services/cambly/ 完了
  title/meta改善: 10件実行

本日のアクション: X部門バッチ補充完了。他は通常運用。
```

社長はこの報告を読んで、問題があれば指示を出す。問題なければ「OK」の一言で済む。

---

## COOのコマンド体系

COOには4つの主要コマンドを持たせます：

### `/coo dashboard` — 全社ダッシュボード

全部門のKPIを一画面で確認するコマンドです：

```
══════════════════════════════════════
  AI会社 経営ダッシュボード
  2026-03-21
══════════════════════════════════════

■ 全社サマリー
  エージェント数: 8名（稼働: 7、停止: 1）
  今月コスト: $1.20 / $5.00（消化率 24%）

■ SEO部門
  部長: seo-bucho [稼働中]
  今月KPI:
    - オーガニッククリック: 1,240 / 2,000（62%）
    - 平均CTR: 2.1% / 3.0%
  パイプライン最終実行: 2026-03-20

■ X部門
  部長: x-bucho [稼働中]
  今月KPI:
    - インプレッション日平均: 5,200 / 8,000（65%）
    - エンゲージメント率: 2.3% / 2.0%（達成!）
    - フォロワー純増: +320 / +500

■ 注意事項
  - SEO部門のCTRが目標を大きく下回っています
    → title/meta改善の頻度を上げることを推奨
══════════════════════════════════════
```

これが「経営ダッシュボード」です。全社の健康状態が一目で分かる。

### `/coo review` — 週次統合レビュー

毎週月曜に実行する統合レビューです：

1. 全部門のKPIトレンドを確認（前週比）
2. 完了/未完了タスクを集計
3. 以下の観点で分析：
   - **達成**: 今週うまくいったこと
   - **課題**: 目標未達の項目と原因
   - **提案**: 来週の優先アクション（最大3つ）
   - **リソース**: 部門間のリソース再配分が必要か
   - **コスト**: 予算消化ペースは適切か

### `/coo kpi` — KPI確認・設定

全社レベルのKPIを一覧表示し、目標の変更もできます：

| カテゴリ | 指標 | 今月実績 | 目標 | 達成率 |
|---------|------|---------|------|--------|
| SEO | オーガニッククリック | 1,240 | 2,000 | 62% |
| SNS | 総インプレッション | 156,000 | 240,000 | 65% |
| コスト | API費用合計 | $1.20 | $5.00 | 24% |

### `/coo cost` — コスト分析

部門別・スキル別のコスト内訳を表示し、予算超過リスクがあれば警告します。Claude Maxを使っていれば基本的にAPIコストはゼロですが、外部API（Search Console API、SNS API等）の利用料や、Haiku/SonnetをAPI経由で使っている場合のコスト管理に必要です。

---

## エスカレーションの設計

COOに全てを任せるわけではありません。「ここから先は社長判断」というラインを明確に引いておきます。

### 部長 → COOのエスカレーション

部長が解決できない問題がCOOに上がってくるパターン：

1. **パイプラインが2回連続FAIL** — COOがログを確認して原因特定
2. **部門間の競合** — API予算や実行時間の配分を調整
3. **KPI大幅未達** — 施策の優先順位を再設定

### COO → 社長のエスカレーション

以下の場合のみ、COOは社長に報告して判断を仰ぎます：

- 月間予算の80%を超過しそうな場合
- 2部門以上で同時にKPI大幅未達（目標の50%以下）
- セキュリティ・コンプライアンス関連の問題
- 新部門・新プロダクトの立ち上げ判断
- COOの権限では判断できない戦略的決定

逆に言えば、**これ以外のことは全てCOOが判断して実行していい**。この「任せる範囲」を明文化しておくことで、社長の負担を最小限にしつつ、AIの暴走を防ぎます。

---

## 権限マトリクス — 誰が何をできるか

組織が大きくなると「誰が何をしていいのか」が曖昧になります。これを防ぐのが権限マトリクスです：

| アクション | COO | 部長 |
|-----------|-----|------|
| 全社KPIの設定・変更 | OK | NG |
| 部門KPIの設定・変更 | OK | OK（自部門のみ） |
| 部門間の優先度調整 | OK | NG |
| 予算の再配分 | OK | NG |
| タスクチケットの作成 | OK | OK（自部門のみ） |
| エージェントの一時停止 | OK | OK（自部門のみ） |
| 社長への直接報告 | OK | NG（COO経由） |
| スキルの修正・更新 | NG（社長の承認必要） | NG |

注目すべきは最後の行です。**スキル（SKILL.md）の修正は社長しかできない**。これはAI組織の安全弁です。AIがAIの「職務記述書」を勝手に書き換えたら、もう人間はコントロールできなくなる。スキルの定義だけは、必ず人間が管理します。

## cronで定期実行する

COOの日次オペレーションを完全自動化するには、cronで定期実行を設定します。

### macOSの場合（launchd）

`~/Library/LaunchAgents/com.ai-company.coo-daily.plist` を作成：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>Label</key>
  <string>com.ai-company.coo-daily</string>
  <key>ProgramArguments</key>
  <array>
    <string>/usr/local/bin/claude</string>
    <string>-p</string>
    <string>/coo</string>
    <string>--dangerously-skip-permissions</string>
  </array>
  <key>WorkingDirectory</key>
  <string>/Users/you/projects/ai-company</string>
  <key>StartCalendarInterval</key>
  <dict>
    <key>Hour</key>
    <integer>7</integer>
    <key>Minute</key>
    <integer>0</integer>
  </dict>
  <key>StandardOutPath</key>
  <string>/tmp/coo-daily.log</string>
  <key>StandardErrorPath</key>
  <string>/tmp/coo-daily-error.log</string>
</dict>
</plist>
```

```bash
launchctl load ~/Library/LaunchAgents/com.ai-company.coo-daily.plist
```

これで毎朝7時にCOOが起動して、全部門の状況確認→判断→実行→報告まで自動で走ります。

### Linuxの場合（crontab）

```bash
0 7 * * * cd /home/you/projects/ai-company && claude -p "/coo" --dangerously-skip-permissions >> /tmp/coo-daily.log 2>&1
```

### 週次レビューの自動化

毎週月曜の朝に `/coo review` を実行する設定も追加しておくと便利です：

```bash
# 毎週月曜 7:30
30 7 * * 1 cd /home/you/projects/ai-company && claude -p "/coo review" --dangerously-skip-permissions >> /tmp/coo-review.log 2>&1
```

## Telegram/Discord連携 — 外出先から経営する

cronでCOOが自動実行されるようになると、次に欲しくなるのが「結果の通知」です。ログファイルを見に行くのは面倒なので、TelegramかDiscordに送らせましょう。

### Telegram Botの設定

1. BotFather（@BotFather）で新しいBotを作成
2. Bot Tokenを取得
3. 自分のChat IDを取得

COOのSKILL.mdに通知セクションを追加：

```markdown
## 報告の送信

日次レポートは以下の方法で社長に送信する:

curl -s -X POST "https://api.telegram.org/bot${TELEGRAM_BOT_TOKEN}/sendMessage" \
  -d "chat_id=${TELEGRAM_CHAT_ID}" \
  -d "text=${REPORT_TEXT}" \
  -d "parse_mode=Markdown"
```

### Discord Webhookの場合

1. Discordサーバーでチャンネルを作成
2. チャンネル設定 → 連携サービス → Webhook → 新しいWebhook
3. Webhook URLをコピー

```bash
curl -s -X POST "${DISCORD_WEBHOOK_URL}" \
  -H "Content-Type: application/json" \
  -d "{\"content\": \"${REPORT_TEXT}\"}"
```

これで、朝起きてスマホを開いたらCOOからの日次レポートが届いている、という体験が実現します。電車の中でレポートを読んで、問題があれば「SEOの優先度を上げて」とメッセージを送り返す。それだけで、AI会社は方向修正されて走り続けます。

## COOを導入する前と後

COOなしの場合：

```
社長 → X部門部長に指示
社長 → SEO部門部長に指示
社長 → 結果を確認
社長 → 部門間の調整
社長 → コスト確認
（毎日30〜60分）
```

COOありの場合：

```
社長: 「おはよう」
COO: （全部やって報告）
社長: 「OK」
（毎日2分）
```

1日30分が2分になる。月換算で約15時間の節約です。しかもCOOは「忘れる」ことがない。人間なら「今日は忙しいからSEOのチェックはスキップ」となりがちですが、COOは毎日同じクオリティで全部門を確認します。

# AI会社 COO（最高執行責任者）

## 役割

社長の右腕として、全部門を横断的に管理する。
部長は自部門の運用に責任を持ち、COOは部門間の調整と全社最適化に責任を持つ。

**判断の基本原則**: 収益に近い施策を優先する。迷ったら社長に聞く。

## 日次オペレーション（社長は「おはよう」だけ）

社長が「おはよう」と声をかけたら、COOが全部門の状況を確認し、
必要な部長にGoサインを出す。社長は個別の部長に指示する必要はない。

### `/coo` or 社長の「おはよう」— 日次オペレーション開始

**Step 1: 状況確認**

各部門の状態を確認する:
- 予約コンテンツの残数
- 前日の実績
- 自動処理（GitHub Actions等）の成否

**Step 2: 判断 → 各部長にGoサイン**

| 状況 | COOの判断 | 部長への指示 |
|------|----------|------------|
| **毎日（必須）** | 前日のパフォーマンス分析 | `/bucho daily` を実行 |
| コンテンツ残 ≤ 6件 | バッチ補充が必要 | `/bucho post` でバッチ作成 |
| 3日以上データ未収集 | データ取得が必要 | `data-collector` を実行 |
| 自動処理に失敗あり | エラー対処が必要 | ログ確認 → 原因特定 → 修正 |

**Step 3: 実行**

判断に基づき、部長スキルを直接呼び出して実行する。
社長への確認は不要（COOに委任されている）。

**Step 4: 報告**

## エスカレーション対応

### 部長からのエスカレーション受付

1. **2回連続FAIL** → ログを確認し、原因を特定
2. **部門間の競合** → リソース配分を調整
3. **KPI大幅未達** → 原因分析し、施策の優先順位を再設定

### 社長へのエスカレーション基準

以下の場合のみ社長に報告する:
- 月間予算の80%を超過しそうな場合
- 2部門以上で同時にKPI大幅未達
- セキュリティ関連の問題
- 新部門・新プロダクトの立ち上げ判断
- COOの権限では判断できない戦略的決定

## 各ディレクトリの詳細

### `skills/` — スキルのソースコード

AI社員の「職務記述書」を格納する場所。1スキル = 1フォルダ = 1つの `SKILL.md`。

**命名規則**: `{部門名}-{役割}`
- 例: `seo-writer`, `sns-bucho`, `blog-checker`

**シンボリックリンクで公開**:
```bash
ln -s ~/projects/claude/ai-company/skills/{スキル名} ~/.claude/skills/{スキル名}
```
これにより、Claude Codeがどのディレクトリからでもスキルを呼び出せるようになる。

### `{部門名}-knowledge/` — 知識ベース

スキルが参照・更新する「組織の記憶」。時間とともに育っていく。

| サブフォルダ | 役割 | 更新頻度 |
|-------------|------|---------|
| `craft/` | テンプレート・ルール・フック | 月次（レビュー時） |
| `voice/` | トーン・口調の定義 | 低頻度（方針変更時） |
| `facts/` | 根拠データ・統計 | 随時（新データ発見時） |
| `themes/` | テーマローテーション | 週次（投稿時） |
| `hypotheses/` | テスト中の仮説 | 週次（レビュー時） |
| `learnings/` | 確認済みパターン | 週次（検証完了時） |
| `posts/` | 出力ログ・スケジュール | 日次（投稿・収集時） |

**INDEX.md の重要性**: 知識ベースが大きくなると、AIが全ファイルを読む必要はなくなる。INDEX.md に「どのタスクのときにどのファイルを読むか」を書いておくことで、効率的に必要な知識だけを参照できる。

### `scripts/` — 自動化スクリプト

API呼び出しやデータ加工など、SKILL.md内で記述しきれない処理を外出しする場所。

### `.github/workflows/` — 定時実行

GitHub Actionsのワークフローファイル。cronで定時実行する自動投稿やデータ収集に使う。

# 付録C：よくある失敗と対策

AI会社を構築・運用する中で実際に遭遇した失敗パターンと、その対策をまとめます。

---

## 1. チェッカーなしで回したら品質崩壊した

### 何が起きたか

「AIは賢いから品質チェックなんていらないだろう」と思い、ライタースキルの出力をそのまま公開に回していた。最初の数回は問題なかったが、1週間もすると:

- 文字数制限を無視した出力が増えた
- 根拠のない数字（「9割の人が〜」）が混入した
- NGワードリストに入っているはずの表現が使われていた
- テンプレート構造を無視した自由形式が出てきた

AIは「やりました」と報告するが、実際にはルールを守っていないことがある。人間のダブルチェックに頼ると、結局自分の時間が取られて自動化の意味がなくなる。

### 対策

**チェッカースキルは初日から必ず入れる。**

チェッカーの設計ポイント:
- 検査項目は「はい/いいえ」で判定できる客観基準にする
- 「良い文章か」ではなく「文字数は130〜140字か」「NGワードを含まないか」
- 重大度をFAIL/WARNINGに分けて、FAILは即差し戻し
- **2回連続NGで部長エスカレーション**のルールを入れる（無限リトライ防止）

チェッカーがあるだけで品質は劇的に安定する。コストは「スキルが1つ増える」だけで、得られる安心感は計り知れない。

---

## 2. スキルが長すぎてコンテキストを圧迫した

### 何が起きたか

「完璧なスキルを作ろう」として、1つのSKILL.mdに以下を全部詰め込んだ:
- 全手順の詳細説明
- あらゆるエッジケースの対処法
- 過去のログや実績データ
- 参考情報のリンク集

結果、SKILL.mdが500行を超え、Claude Codeのコンテキストウィンドウの大部分をスキルの読み込みだけで消費してしまった。肝心の作業に使えるコンテキストが足りなくなり、指示の後半部分が無視される現象が頻発した。

### 対策

**SKILL.mdは「職務記述書」であり「マニュアル全集」ではない。**

実践ルール:
- SKILL.mdは200行以内を目標にする
- 詳細データは知識ベース（knowledge/）に外出しして、必要な時だけ読み込む
- INDEX.mdに「このタスクではこのファイルを読め」とナビゲーションを書く
- エッジケースは「よくある3パターン」だけ書き、それ以外は「部長にエスカレーション」で逃がす

**知識ベースの活用パターン**:
```
SKILL.md: 「Step 3でfacts/から根拠データを選定する」（1行）
facts/INDEX.md: 全Factの一覧（必要な時だけ読む）
facts/テーマ名.md: 実データ（使う時だけ読む）
```

こうすれば、SKILL.md自体はコンパクトに保ちつつ、必要な情報には常にアクセスできる。

---

## 3. 部長が勝手に判断しすぎて暴走した

### 何が起きたか

部長スキルに「状況を見て最適な判断をせよ」と書いたら、以下のようなことが起きた:

- 週次レビューで「パフォーマンスが低いのでテンプレートを全面刷新します」と勝手に全テンプレートを書き換えた
- 「このテーマは効果が低い」と判断して、4テーマを一気にローテーションから外した
- 根拠が薄い「パターン」を大量にpatterns.mdに登録し、知識ベースがノイズだらけになった

部長に「判断力」を持たせること自体は正しい。問題は、判断の**範囲と条件**を明確にしなかったこと。

### 対策

**部長の権限に「ガードレール」を設ける。**

具体的なルール:
1. **サンプル数の最低基準**: 「5件未満の傾向をパターンとして登録してはいけない」
2. **変更の上限**: 「1回のレビューでテンプレートの変更は最大2件まで」
3. **不可逆な変更は社長承認**: 「テーマの削除」「テンプレートの全面刷新」は社長の承認を得る
4. **仮説 → 検証 → パターン**: いきなりパターン登録せず、まず仮説（hypotheses/active.md）として登録し、十分なデータで検証してからpatterns.mdに昇格させる

部長スキルのSKILL.mdに「権限と制約」セクションを設け、「できること/できないこと」を明示するのが最も効果的だった。

---

## 4. メモリと知識ベースがごちゃ混ぜになった

### 何が起きたか

Claude Codeのメモリシステム（`~/.claude/memory/`）と、AI会社の知識ベース（`knowledge/`）の区別がつかなくなった。

- 投稿の勝ちパターンがメモリに書かれていて、スキルから参照できない
- 社長の好み（「この表現は使わないで」）が知識ベースに書かれていて、別プロジェクトで参照できない
- 同じ情報が両方に書かれていて、片方だけ更新されて矛盾が発生

### 対策

**明確な住み分けルールを作る。**

| 情報の種類 | 保存先 | 理由 |
|-----------|-------|------|
| 投稿の勝ちパターン | knowledge/learnings/ | スキルが業務で使う知識 |
| 社長の好み・方針 | ~/.claude/memory/ | 全プロジェクトで参照する個人設定 |
| テンプレート・フック | knowledge/craft/ | 部門固有の業務ツール |
| APIキーの場所 | ~/.claude/memory/ | 環境情報（プロジェクト横断） |
| 仮説・検証結果 | knowledge/hypotheses/ | データドリブンな業務知識 |

判断基準は単純: **「このスキルが仕事で使うか？」→ knowledge/。「社長個人の情報か？」→ memory/**。

CLAUDE.mdにこのルールを明記しておけば、スキルが勝手にメモリに業務知識を書き込むことはなくなる。

## 5. コストが想定外に膨らんだ

### 何が起きたか

Claude Max（$100/月のサブスクリプション）で運用していたが、以下の理由でAPI併用が必要になり、コストが膨らんだ:

- GitHub Actionsでの定時実行にはAPIが必要（Claude Maxはインタラクティブ利用のみ）
- パイプラインの各ステップでサブエージェントを生成すると、1回のパイプラインで大量のトークンを消費
- 知識ベースが大きくなるにつれ、読み込むだけで相当のトークンを使う
- デバッグ中の試行錯誤で無駄なAPI呼び出しが増えた

### 対策

**コスト最適化の3つの柱:**

#### (1) Claude Maxでできることを最大化する

Claude Max（$100/月）はインタラクティブ利用なら実質無制限。日次オペレーション（「おはよう」→ COOが全部回す）はClaude Maxの範囲内で完結させる。

#### (2) API利用はHaikuを基本にする

定時実行が必要な処理（GitHub Actionsでの自動投稿など）はAPIを使うが、モデル選択を工夫する:

| 処理 | モデル | 理由 |
|------|-------|------|
| 定型的な投稿（テンプレート適用） | Haiku | 判断不要、パターンマッチのみ |
| データ収集・CSV加工 | Haiku | 手順通りの実行、判断不要 |
| 品質チェック | Sonnet | チェック項目の判定に中程度の推論が必要 |
| 週次レビュー・戦略判断 | Opus（Max） | 高度な分析・判断が必要 |

#### (3) 知識ベースの読み込みを最適化する

- INDEX.mdで「今のタスクに必要なファイルだけ」を指定する
- 全ファイル一括読み込みをやめる
- factsやpostsなど大きなファイルは、必要な行だけ読む

**実際の月間コスト例**:

| 項目 | コスト |
|------|-------|
| Claude Max（日次オペレーション・週次レビュー） | $100/月（固定） |
| API: Haiku（自動投稿 × 60回/月） | $1〜2/月 |
| API: Sonnet（品質チェック × 30回/月） | $3〜5/月 |
| Ahrefs Lite（SEOデータ） | 約$150/月 |
| **合計** | **約$255〜260/月** |

外部ツール（Ahrefs等）を除けば、AI会社の運用コストは月$105程度。人を1人雇うことを考えれば圧倒的に安い。

---

## まとめ：失敗から学んだ5つの原則

1. **チェッカーは初日から入れる** — 品質は仕組みで守る
2. **SKILL.mdは短く、知識は外に出す** — コンテキストは有限資源
3. **部長には権限の上限を設ける** — 自由すぎる判断は暴走する
4. **メモリと知識ベースは明確に分ける** — 混ぜると矛盾が起きる
5. **コストは設計段階で最適化する** — 後から削るのは難しい

## Claude Max $100/月でどこまでできるか

Claude Max（$100/月）は、Claude Codeをインタラクティブに使う場合の定額プラン。対話的な操作であれば実質無制限に使える。

### Claude Maxの範囲内でできること

| 業務 | 頻度 | 説明 |
|------|------|------|
| 日次オペレーション | 毎日 | 「おはよう」→ COOが全部門の状況確認・部長への指示・実行・報告 |
| 週次レビュー | 週1回 | データ分析・パターン発見・知識ベース更新・方針決定 |
| コンテンツ作成 | 随時 | ライター→チェッカー→承認のフロー |
| パイプライン実行 | 週1回 | データ収集→分析→実行の3ステップ連鎖 |
| KPI確認 | 随時 | ダッシュボード表示・トレンド分析 |
| スキルの改善 | 随時 | SKILL.mdの修正・知識ベースの整理 |

**つまり、人間が画面の前にいて「おはよう」「レビューして」「投稿作って」と指示を出す作業は全てClaude Maxの範囲内。** これだけで小規模なAI会社は十分に回せる。

### Claude Maxだけでは足りないケース

- **定時自動実行**: GitHub Actionsやcronで人間不在で動かす処理にはAPIが必要
- **外部トリガー**: Webhookやスケジューラーからの自動起動

## API併用のコスト最適化

Claude Maxで対応できない「無人自動実行」にはAPIを使う。ここでのコスト削減が重要。

### モデル選択の指針

| 処理の性質 | 推奨モデル | 1回あたりコスト目安 | 理由 |
|-----------|-----------|-------------------|------|
| 手順通りの実行（テンプレート適用、CSV加工） | Haiku | $0.001〜0.01 | 判断不要。パターンマッチのみ |
| 品質チェック（合否判定） | Haiku〜Sonnet | $0.005〜0.05 | 基準が明確ならHaikuで十分 |
| 文章生成（投稿文の作成） | Sonnet | $0.02〜0.10 | 品質が必要だがOpus不要 |
| 高度な分析・判断 | Opus（Max利用） | $0（Max内） | 対話的に実行すればMaxの範囲 |

### コスト削減の具体テクニック

#### 1. 定型処理はHaikuに任せる

自動投稿のように「schedule.jsonから次の投稿を読んでXに投稿する」だけの処理は、Haikuで十分。判断が入らない定型作業にOpusやSonnetを使うのは無駄。

#### 2. 知識ベースの読み込みを最小化する

API呼び出しのトークン消費で最も大きいのは、入力（プロンプト + 知識ベースの読み込み）。INDEX.mdで「このタスクにはこのファイルだけ読め」と指定することで、不要なファイルを読み込まない。

#### 3. チェッカーの基準を明確にする

チェック項目が「文字数は130〜140字か」「NGワードを含まないか」のように明確であれば、Haikuでも正確に判定できる。「良い文章か」のような曖昧な基準にすると、高価なモデルが必要になる。

#### 4. バッチ処理でAPI呼び出し回数を減らす

投稿を1件ずつ作るのではなく、7日分14投稿を1回のAPI呼び出しでバッチ作成する。API呼び出しのオーバーヘッドが1回分で済む。

## コストを抑えるための設計原則

### 原則1: 「判断」はMaxで、「実行」はAPIで

高度な判断（戦略決定、パターン分析、方針変更）はClaude Max内でインタラクティブに行う。定型的な実行（投稿、データ加工、チェック）だけをAPI経由で自動化する。

### 原則2: API呼び出しは「最安モデルで動くか？」から考える

新しい自動化を追加するとき、まずHaikuで試す。Haikuで品質が足りなければSonnetに上げる。Opusが必要な処理は、そもそもAPIではなくMaxの対話で行うべきケースが多い。

### 原則3: コスト追跡をCOOに任せる

COOスキルに `/coo cost` コマンドを持たせ、月次のAPI消費量を自動チェックさせる。予算の80%に達したらアラートを出す仕組みにしておけば、知らない間にコストが膨らむことを防げる。
