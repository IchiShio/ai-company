# 第2章：最小構成で始める — 最初のスキルを作る

前章で「AI会社」の設計思想を理解しました。ここからはいよいよ実践です。

最初にやることは、**たった1つのスキルを作る**こと。いきなり組織全体を設計する必要はありません。まずは「社員1人を雇う」ところから始めましょう。

---

## SKILL.mdとは何か

Claude Codeのスキルとは、`~/.claude/skills/` に配置された **SKILL.md** ファイルのことです。

普通のプロンプトとの違いは明確です。プロンプトは「1回きりの指示」ですが、SKILL.mdは**職務記述書**です。名前があり、役割があり、手順が定義されている。何度呼び出しても同じ品質で仕事をしてくれる「再現可能な社員」を作るのがSKILL.mdの目的です。

---

## SKILL.mdの基本構造

SKILL.mdは大きく2つのパーツで構成されています。

### 1. frontmatter（ヘッダー情報）

ファイルの先頭に `---` で囲まれたYAML形式のメタデータを書きます。

```markdown
---
name: article-writer
description: |
  ブログ記事のライター。指定テーマで記事を執筆する。

  次のような依頼で使用すること：
  - 「記事を書いて」「ブログ記事を作成して」
  - 「○○について記事にして」
---
```

**name** はスキルの識別子です。フォルダ名と一致させるのがおすすめです。

**description** は2つの役割を持っています。

1つ目は「このスキルが何をするか」の説明。2つ目が重要で、**「どんな依頼のときにこのスキルを呼び出すべきか」の判定基準**です。Claude Codeは `description` を読んで、ユーザーの依頼にどのスキルが適切かを判断します。だから「次のような依頼で使用すること」というリストを入れておくと、意図通りに呼び出されやすくなります。

### 2. 本文（職務記述書）

frontmatterの下に、Markdown形式で仕事の手順を書きます。ここが「社員への業務マニュアル」です。

```markdown
# 記事ライター

## 役割
指定されたテーマについて、ターゲット読者に合わせた記事を執筆する。

## 入力
呼び出し時に以下を受け取る：
- テーマ（必須）
- ターゲット読者（任意、デフォルト: 一般読者）
- 文字数目安（任意、デフォルト: 3,000文字）

## Step 1: リサーチ
...

## Step 2: 構成作成
...
```

ポイントは、**曖昧さを排除する**ことです。

「いい感じに書いて」ではなく、「Step 1で何をして、Step 2で何をして、最終的にどういう形式で出力するか」を明確に書く。人間の新入社員に渡す業務マニュアルを想像してください。そのレベルの具体性が必要です。

---

## 実例：「記事ライター」スキルを作ってみよう

では、実際に1つスキルを作ります。ブログ記事を書いてくれる「記事ライター」を雇いましょう。

### ステップ1：フォルダを作る

スキルは `~/.claude/skills/スキル名/SKILL.md` というパスに配置します。

```bash
mkdir -p ~/.claude/skills/article-writer
```

### ステップ2：SKILL.mdを書く

以下の内容で `~/.claude/skills/article-writer/SKILL.md` を作成します。

```markdown
---
name: article-writer
description: |
  ブログ記事のライター。指定されたテーマで読者に役立つ記事を執筆する。

  次のような依頼で使用すること：
  - 「記事を書いて」「ブログ記事を作成して」
  - 「○○について記事にして」
  - パイプラインの記事作成ステップとして自動呼び出し
---

# 記事ライター

## 役割

指定されたテーマについて、ターゲット読者に最適化された記事を執筆する。
自分では公開しない（完成した記事をユーザーまたは後続のチェッカーに渡す）。

---

## 入力

呼び出し時に以下の情報を受け取る：

| 項目 | 必須/任意 | デフォルト |
|------|-----------|------------|
| テーマ | 必須 | — |
| ターゲット読者 | 任意 | 一般読者 |
| 文字数目安 | 任意 | 3,000文字 |
| トーン | 任意 | です・ます調 |
| 出力形式 | 任意 | Markdown |

---

## Step 1: リサーチ

テーマに関する情報を収集する。

1. 指定されたテーマのキーワードで検索し、上位記事の構成を確認する
2. 読者が知りたいであろう疑問点を3〜5個リストアップする
3. 記事に盛り込むべきデータ・事例を収集する

---

## Step 2: 構成作成

記事の骨格を作る。

1. タイトル案（2〜3案）
2. リード文（100〜150文字）
3. H2見出しの構成（5〜7セクション）
4. 各セクションで書く内容のメモ（箇条書き）

---

## Step 3: 執筆

構成に沿って本文を書く。

### 品質基準
- 文字数: 指定文字数の±10%以内
- 各H2セクション: 最低300文字
- 具体例・データ: 各セクションに最低1つ
- 読みやすさ: 1文50文字以内を目安

### 禁止事項
- 根拠のない数字（「9割の人が〜」等）
- 他サイトのコピー
- 事実と異なる主張

---

## Step 4: セルフチェック

提出前に以下を自己確認する。

- [ ] 文字数は基準範囲内か
- [ ] タイトルとリード文は内容を正確に反映しているか
- [ ] 論理の飛躍がないか
- [ ] 誤字脱字がないか
- [ ] 禁止事項に違反していないか

---

## 出力フォーマット

```
📝 記事（テーマ: {テーマ名}）

ターゲット: {読者層}
文字数: {実際の文字数}
構成: H2 × {セクション数}

---
{記事本文（Markdown形式）}
---

セルフチェック結果:
  文字数: ✅ / ❌
  具体例: ✅ / ❌
  禁止事項: ✅ / ❌
```

---

## 注意事項

- 最終的な公開判断はユーザー（社長）が行う。勝手に公開しない
- チェッカースキルが存在する場合は、チェッカーの判定を待つ
- 不明な点があれば、作業を進める前にユーザーに確認する
```

### ステップ3：Claude Codeで呼び出す

スキルを `~/.claude/skills/` に配置したら、Claude Codeを起動するだけでスキルは自動的に認識されます。特別なコマンドは不要です。

```
> SEOについて初心者向けの記事を書いて
```

Claude Codeは `description` の内容を読んで、ユーザーの依頼に適したスキルを自動で選択します。上の依頼なら「記事を書いて」がdescriptionの呼び出し条件にマッチするため、article-writerのSKILL.mdが読み込まれ、そこに書かれた手順通りに仕事を進めてくれます。

だからこそ、前節で解説した `description` の「次のような依頼で使用すること」リストが重要なのです。このリストが曖昧だと、意図しないスキルが呼ばれたり、呼ばれるべきスキルが無視されたりします。

---

## スキルを「管理」する：リポジトリ管理のすすめ

スキルが増えてくると、ファイル管理が重要になります。おすすめの方法は、**Gitリポジトリでスキルを一元管理し、シンボリックリンクで公開する**パターンです。

```
# リポジトリ側（ソース）
my-ai-company/
└── skills/
    ├── article-writer/
    │   └── SKILL.md
    ├── article-checker/
    │   └── SKILL.md
    └── data-collector/
        └── SKILL.md

# Claude Codeが読み込む場所（シンボリックリンク）
~/.claude/skills/
├── article-writer -> ~/projects/my-ai-company/skills/article-writer
├── article-checker -> ~/projects/my-ai-company/skills/article-checker
└── data-collector -> ~/projects/my-ai-company/skills/data-collector
```

シンボリックリンクの作成コマンド：

```bash
ln -s ~/projects/my-ai-company/skills/article-writer ~/.claude/skills/article-writer
```

こうすることで、スキルの変更履歴がGitで追跡でき、複数マシンでも同じ組織構成を再現できます。

## SKILL.mdを書くときの5つのコツ

実際に何十ものスキルを運用してきた経験から、効果的なSKILL.mdを書くコツをまとめます。

### 1. Stepを明示する

「いい感じにやって」ではAIは手を抜きます。Step 1、Step 2...と明確に段階を分けることで、各ステップを確実に実行させられます。

### 2. 出力フォーマットを固定する

出力の形式が毎回違うと、後続のチェッカーやパイプラインが処理できません。テンプレートを定義して、常に同じ形式で出力させましょう。

### 3. 「禁止事項」を書く

AIは指示されていないことを勝手にやりがちです。「やるべきこと」だけでなく「やってはいけないこと」を明記するのが非常に効果的です。

### 4. descriptionに呼び出し条件を書く

frontmatterの `description` に「次のような依頼で使用すること」と具体例を列挙しておくと、意図しないスキルが呼ばれる問題を防げます。

### 5. 1スキル1責務を徹底する

「記事を書いてチェックもして公開もする」を1つのスキルに詰め込まないでください。書く人、チェックする人、公開する人は別のスキルにする。これが品質を保つ最大の秘訣です。

## やってみよう：あなたの業務で1つスキルを作る

ここまで読んだら、実際に手を動かしてみましょう。

1. あなたの業務で「毎回同じ手順でやっている作業」を1つ選ぶ
2. その手順をStep 1, 2, 3...と書き出す
3. SKILL.mdのテンプレートに当てはめる
4. `~/.claude/skills/` に配置して呼び出してみる

最初のスキルが動いた瞬間、「これは社員だ」と実感するはずです。

ただし、ここで1つ重大な問題があります。作ったスキルに仕事をさせてみると、**AIは「やりました」と報告しておきながら、実は手を抜いている**ことがあるのです。

次の章では、この問題を解決する「チェッカースキル」を作ります。
