# 第7章：全社を統括する — COOと経営ダッシュボード

## 朝起きたらレポートが届いていた

ある朝、スマホを開いたらTelegramに通知が来ていました。

```
おはようございます。本日の状況です。

■ X部門
  投稿ストック: 残8件（3/24まで）→ 問題なし
  GitHub Actions: 全件成功
  昨日のパフォーマンス:
    朝投稿: imp 3,500 / eng 2.1%
    夕投稿: imp 2,800 / eng 1.8%

■ SEO部門
  本日のリライト対象: /services/cambly/
  title/meta改善: 10件完了
  前日比: クリック +12、インプレ +340

本日のアクション: 全部門通常運用。異常なし。
```

これを送ってきたのは、僕が雇ったCOO（最高執行責任者）です。もちろんAIの。

COOは毎朝、全部門の状況を確認して、部長たちにGoサインを出し、その結果を僕に報告してくれます。僕がやることは「おはよう」と声をかけることだけ。あとはCOOが全部回してくれます。

---

## COOの役割 — 部長の上に立つ存在

第5章で作った「部長」は、自分の部門を運用する責任者でした。X部門ならX部門、SEO部門ならSEO部門。でも部長には「部門間の調整」ができません。

- 「今月はSEOのほうが優先度高いから、X部門のAPI予算を少し回そう」
- 「2部門とも同時にKPI未達だ。どちらから手をつけるべきか」
- 「全社のコストが予算の80%に達している。ペースを落とすべきか」

こういう判断ができるのがCOOです。

組織の階層を整理するとこうなります：

```
社長（人間・あなた）
  └── COO（全社統括）
        ├── X部門 部長
        │     ├── ライタースキル
        │     ├── チェッカースキル
        │     └── データ収集スキル
        └── SEO部門 部長
              ├── ライタースキル
              ├── チェッカースキル
              └── データ収集スキル
```

社長が直接やり取りするのはCOOだけ。COOが部長に指示を出し、部長が社員スキルを使って実行する。

---

## COOスキルの設計

COOのSKILL.mdは、これまでのスキルより一段複雑です。でも構造は同じ。「役割」「管轄」「コマンド」「エスカレーション基準」を定義します。

```markdown
---
name: coo
description: |
  AI会社の最高執行責任者。全部門横断の経営管理を担当。
  KPI集約、予算・コスト追跡、部門間優先度調整、
  社長への統合レポート作成を行う。
---

# COO（最高執行責任者）

## 役割
社長の右腕として、全部門を横断的に管理する。
部長は自部門の運用に責任を持ち、COOは部門間の調整と全社最適化に責任を持つ。

**判断の基本原則**: 収益に近い施策を優先する。迷ったら社長に聞く。

## 管轄部門
| 部門 | 部長スキル | 主な役割 |
|------|-----------|---------|
| SEO部門 | seo-bucho | Webサイトの検索最適化・コンテンツ改善 |
| X部門 | x-bucho | SNSアカウントの運用・成長 |
```

ポイントは **「判断の基本原則」** です。COOは判断する権限を持つので、判断基準を明文化しておかないとAIが暴走します。「収益に近い施策を優先」「迷ったら社長に聞く」——このシンプルな基準が、COOの全ての判断の土台になります。

---

## 日次オペレーション — 「おはよう」で始まる1日

COOの最も重要な仕事は日次オペレーションです。毎朝の流れはこうなります：

### Step 1: 状況確認

COOがまず全部門の現在地を確認します：

- **投稿ストック残数**: schedule.jsonの未投稿件数を数える
- **GitHub Actions**: 自動投稿ワークフローの直近の成功/失敗を確認
- **前日のパフォーマンス**: 投稿のimp、エンゲージメント率を取得

### Step 2: 判断

状況に応じてCOOが判断を下します。ここがただのスクリプトとの違いです：

| 状況 | COOの判断 |
|------|----------|
| ストック残 <= 6件 | 部長にバッチ補充を指示 |
| 3日以上データ未収集 | データ収集スキルを実行 |
| 投稿データが10件以上 | 部長に週次レビューを指示 |
| GitHub Actions 失敗 | ログ確認 → 原因特定 → 修正 |
| 全て正常 | 日次タスクを部長に指示 |

### Step 3: 実行

判断に基づいて、部長スキルを呼び出します。COOは自分では「作業」をしません。部長に指示を出すだけ。部長がさらに社員スキルを使って実行する。

### Step 4: 報告

全部門の結果を統合して、社長に報告します。

```
■ X部門
  投稿ストック: 残4件 → バッチ補充しました（3/30まで延長）
  GitHub Actions: 全件成功

■ SEO部門
  本日のリライト: /services/cambly/ 完了
  title/meta改善: 10件実行

本日のアクション: X部門バッチ補充完了。他は通常運用。
```

社長はこの報告を読んで、問題があれば指示を出す。問題なければ「OK」の一言で済む。

## COOのコマンド体系

COOには4つの主要コマンドを持たせます：

### `/coo dashboard` — 全社ダッシュボード

全部門のKPIを一画面で確認するコマンドです：

```
══════════════════════════════════════
  AI会社 経営ダッシュボード
  2026-03-21
══════════════════════════════════════

■ 全社サマリー
  エージェント数: 8名（稼働: 7、停止: 1）
  今月コスト: $1.20 / $5.00（消化率 24%）

■ SEO部門
  部長: seo-bucho [稼働中]
  今月KPI:
    - オーガニッククリック: 1,240 / 2,000（62%）
    - 平均CTR: 2.1% / 3.0%
  パイプライン最終実行: 2026-03-20

■ X部門
  部長: x-bucho [稼働中]
  今月KPI:
    - インプレッション日平均: 5,200 / 8,000（65%）
    - エンゲージメント率: 2.3% / 2.0%（達成!）
    - フォロワー純増: +320 / +500

■ 注意事項
  - SEO部門のCTRが目標を大きく下回っています
    → title/meta改善の頻度を上げることを推奨
══════════════════════════════════════
```

これが「経営ダッシュボード」です。全社の健康状態が一目で分かる。

### `/coo review` — 週次統合レビュー

毎週月曜に実行する統合レビューです：

1. 全部門のKPIトレンドを確認（前週比）
2. 完了/未完了タスクを集計
3. 以下の観点で分析：
   - **達成**: 今週うまくいったこと
   - **課題**: 目標未達の項目と原因
   - **提案**: 来週の優先アクション（最大3つ）
   - **リソース**: 部門間のリソース再配分が必要か
   - **コスト**: 予算消化ペースは適切か

### `/coo kpi` — KPI確認・設定

全社レベルのKPIを一覧表示し、目標の変更もできます：

| カテゴリ | 指標 | 今月実績 | 目標 | 達成率 |
|---------|------|---------|------|--------|
| SEO | オーガニッククリック | 1,240 | 2,000 | 62% |
| SNS | 総インプレッション | 156,000 | 240,000 | 65% |
| コスト | API費用合計 | $1.20 | $5.00 | 24% |

### `/coo cost` — コスト分析

部門別・スキル別のコスト内訳を表示し、予算超過リスクがあれば警告します。Claude Maxを使っていれば基本的にAPIコストはゼロですが、外部API（Search Console API、SNS API等）の利用料や、Haiku/SonnetをAPI経由で使っている場合のコスト管理に必要です。

## エスカレーションの設計

COOに全てを任せるわけではありません。「ここから先は社長判断」というラインを明確に引いておきます。

### 部長 → COOのエスカレーション

部長が解決できない問題がCOOに上がってくるパターン：

1. **パイプラインが2回連続FAIL** — COOがログを確認して原因特定
2. **部門間の競合** — API予算や実行時間の配分を調整
3. **KPI大幅未達** — 施策の優先順位を再設定

### COO → 社長のエスカレーション

以下の場合のみ、COOは社長に報告して判断を仰ぎます：

- 月間予算の80%を超過しそうな場合
- 2部門以上で同時にKPI大幅未達（目標の50%以下）
- セキュリティ・コンプライアンス関連の問題
- 新部門・新プロダクトの立ち上げ判断
- COOの権限では判断できない戦略的決定

逆に言えば、**これ以外のことは全てCOOが判断して実行していい**。この「任せる範囲」を明文化しておくことで、社長の負担を最小限にしつつ、AIの暴走を防ぎます。

---

## 権限マトリクス — 誰が何をできるか

組織が大きくなると「誰が何をしていいのか」が曖昧になります。これを防ぐのが権限マトリクスです：

| アクション | COO | 部長 |
|-----------|-----|------|
| 全社KPIの設定・変更 | OK | NG |
| 部門KPIの設定・変更 | OK | OK（自部門のみ） |
| 部門間の優先度調整 | OK | NG |
| 予算の再配分 | OK | NG |
| タスクチケットの作成 | OK | OK（自部門のみ） |
| エージェントの一時停止 | OK | OK（自部門のみ） |
| 社長への直接報告 | OK | NG（COO経由） |
| スキルの修正・更新 | NG（社長の承認必要） | NG |

注目すべきは最後の行です。**スキル（SKILL.md）の修正は社長しかできない**。これはAI組織の安全弁です。AIがAIの「職務記述書」を勝手に書き換えたら、もう人間はコントロールできなくなる。スキルの定義だけは、必ず人間が管理します。

## cronで定期実行する

COOの日次オペレーションを完全自動化するには、cronで定期実行を設定します。

### macOSの場合（launchd）

`~/Library/LaunchAgents/com.ai-company.coo-daily.plist` を作成：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>Label</key>
  <string>com.ai-company.coo-daily</string>
  <key>ProgramArguments</key>
  <array>
    <string>/usr/local/bin/claude</string>
    <string>-p</string>
    <string>/coo</string>
    <string>--dangerously-skip-permissions</string>
  </array>
  <key>WorkingDirectory</key>
  <string>/Users/you/projects/ai-company</string>
  <key>StartCalendarInterval</key>
  <dict>
    <key>Hour</key>
    <integer>7</integer>
    <key>Minute</key>
    <integer>0</integer>
  </dict>
  <key>StandardOutPath</key>
  <string>/tmp/coo-daily.log</string>
  <key>StandardErrorPath</key>
  <string>/tmp/coo-daily-error.log</string>
</dict>
</plist>
```

```bash
launchctl load ~/Library/LaunchAgents/com.ai-company.coo-daily.plist
```

これで毎朝7時にCOOが起動して、全部門の状況確認→判断→実行→報告まで自動で走ります。

### Linuxの場合（crontab）

```bash
0 7 * * * cd /home/you/projects/ai-company && claude -p "/coo" --dangerously-skip-permissions >> /tmp/coo-daily.log 2>&1
```

### 週次レビューの自動化

毎週月曜の朝に `/coo review` を実行する設定も追加しておくと便利です：

```bash
# 毎週月曜 7:30
30 7 * * 1 cd /home/you/projects/ai-company && claude -p "/coo review" --dangerously-skip-permissions >> /tmp/coo-review.log 2>&1
```

## Telegram/Discord連携 — 外出先から経営する

cronでCOOが自動実行されるようになると、次に欲しくなるのが「結果の通知」です。ログファイルを見に行くのは面倒なので、TelegramかDiscordに送らせましょう。

### Telegram Botの設定

1. BotFather（@BotFather）で新しいBotを作成
2. Bot Tokenを取得
3. 自分のChat IDを取得

COOのSKILL.mdに通知セクションを追加：

```markdown
## 報告の送信

日次レポートは以下の方法で社長に送信する:

curl -s -X POST "https://api.telegram.org/bot${TELEGRAM_BOT_TOKEN}/sendMessage" \
  -d "chat_id=${TELEGRAM_CHAT_ID}" \
  -d "text=${REPORT_TEXT}" \
  -d "parse_mode=Markdown"
```

### Discord Webhookの場合

1. Discordサーバーでチャンネルを作成
2. チャンネル設定 → 連携サービス → Webhook → 新しいWebhook
3. Webhook URLをコピー

```bash
curl -s -X POST "${DISCORD_WEBHOOK_URL}" \
  -H "Content-Type: application/json" \
  -d "{\"content\": \"${REPORT_TEXT}\"}"
```

これで、朝起きてスマホを開いたらCOOからの日次レポートが届いている、という体験が実現します。電車の中でレポートを読んで、問題があれば「SEOの優先度を上げて」とメッセージを送り返す。それだけで、AI会社は方向修正されて走り続けます。

## COOを導入する前と後

COOなしの場合：

```
社長 → X部門部長に指示
社長 → SEO部門部長に指示
社長 → 結果を確認
社長 → 部門間の調整
社長 → コスト確認
（毎日30〜60分）
```

COOありの場合：

```
社長: 「おはよう」
COO: （全部やって報告）
社長: 「OK」
（毎日2分）
```

1日30分が2分になる。月換算で約15時間の節約です。しかもCOOは「忘れる」ことがない。人間なら「今日は忙しいからSEOのチェックはスキップ」となりがちですが、COOは毎日同じクオリティで全部門を確認します。

## ここまでのまとめ — AI会社の全体像

第1章から第7章まで、AI会社の全体像が見えてきました。最後に整理しておきます。

```
ai-company/
├── skills/
│   ├── coo/SKILL.md              # 最高執行責任者
│   ├── x-bucho/SKILL.md          # X部門 部長
│   ├── x-writer/SKILL.md         # X部門 ライター
│   ├── x-checker/SKILL.md        # X部門 チェッカー
│   ├── x-data-collector/SKILL.md # X部門 データ収集
│   ├── seo-bucho/SKILL.md        # SEO部門 部長
│   ├── seo-writer/SKILL.md       # SEO部門 ライター
│   └── seo-checker/SKILL.md      # SEO部門 チェッカー
├── x-knowledge/                   # X部門 知識ベース
│   ├── INDEX.md
│   ├── craft/
│   ├── facts/
│   ├── hypotheses/
│   ├── learnings/
│   ├── posts/
│   └── themes/
├── seo-knowledge/                 # SEO部門 知識ベース
│   └── ...
└── CLAUDE.md
```

社員が実行し、チェッカーが品質を守り、部長が判断し、知識ベースが学び、COOが統括する。この5層構造が、1人でAI会社を経営するための全体設計です。

付録では、ここまで登場したスキルのテンプレートをそのまま使える形で提供します。コピペして、自分の業務に合わせてカスタマイズしてください。
