# 第5章：部長を雇う — 判断できるAIを作る

前章でパイプラインを作りました。データ収集→分析→実行が自動で流れる。チェッカーが品質を検査する。素晴らしい。

でも、こんな場面が必ず来ます。

「データソースAが落ちている。データBだけで続行するか、全部止めるか？」

チェッカーは「FAILです」と言うだけ。社員スキルは手順通りに動くだけ。**誰も判断してくれない。**

結局あなた（社長）がターミナルに戻ってきて、状況を見て、指示を出すことになる。これでは「自動化」ではなく「半自動化」です。

この章では、**状況を判断して自分で対処できるAI——「部長」**の作り方を解説します。

---

## 社員と部長の決定的な違い

AI会社の3層構造を思い出してください。

```
社長（人間）
  ↓ 例外中の例外のみ介入
部長スキル（判断者）
  ↓ 社員に指示、問題を対処
社員スキル（実行者）
  ↓ 手順通りに実行
```

社員と部長の違いは、一言でいうと**「手順か判断か」**です。

| | 社員スキル | 部長スキル |
|---|---|---|
| 動き方 | 与えられた手順を忠実に実行 | 状況を見て何をすべきか決める |
| 失敗時 | 1回リトライ → 上に報告 | 根本原因を推定し、代替手段を採用 |
| 権限 | 自分の担当工程のみ | 配下の社員全員を指揮 |
| 知識 | SKILL.md内の手順 | 部門の目標・KPI・対処パターン |
| 呼ばれ方 | パイプラインから自動で | エスカレーション時 or コマンドで |

具体例で考えましょう。SEOパイプラインでGA4のデータが取れなかった場合：

**社員（チェッカー）の反応:**
> Check 1 FAIL — ga4_traffic.csv が存在しません。リトライします。
> ……再FAIL。エスカレーションします。

**部長の反応:**
> GA4データは取れていないが、AhrefsデータとSearch Consoleデータは取れている。
> GA4なしでもキーワード分析とバックリンク分析は実行可能。
> → analyzerに「GA4なし」を伝えて、利用可能なデータだけで分析を続行する。
> → 完了後レポートに「GA4データ未取得のため一部分析はスキップ」と警告を記載する。

この違いわかりますか？ 社員は「取れない→ダメ」で止まる。部長は「取れないけど、ここまではできる」と判断して進める。

## 部長スキルの設計パターン

部長スキルのSKILL.mdは、社員スキルとは構造が大きく異なります。

### 社員スキルの構造（おさらい）

```markdown
# 記事ライター

## 入力
- テーマ、ターゲット読者、文字数

## 手順
1. テーマをリサーチする
2. アウトラインを作る
3. 本文を書く
4. 校正する

## 出力
- Markdown記事
```

手順が明確。入力→処理→出力が一直線。

### 部長スキルの構造

```markdown
# SEO部門 部長AI

## 部門の目標
アフィリエイト収入の増加。検索順位改善はその手段。

## 役割
社員スキルは「決められた手順を実行する」のみ。
部長は「状況を判断して何をすべきか決める」。

## 管轄する社員
- data-collector（データ収集）
- analyzer（分析）
- executor（実行）
- checker（品質検査）

## 対処パターン一覧
### パターン1: データ収集の一部失敗
### パターン2: 分析の品質不足
### パターン3: 実行時のエラー
### パターン4: 同じ問題の繰り返し（慢性障害）
### パターン5: 部長でも解決できない場合

## KPI管理
## 定期レビュー
## コマンド一覧
```

手順ではなく**判断基準**が書かれています。「こういう状況ならこう判断する」のパターン集です。

---

## 部長に持たせる4つの権限

部長スキルには、社員にはない権限を持たせます。

### 1. 社員スキルの呼び出し（Agent tool）

部長は配下の社員を自由に呼び出せます。

```markdown
## 管轄する社員

| 社員 | 役割 |
|---|---|
| writer | 投稿ライター |
| schedule-post | 予約投稿 |
| data-collector | データ収集 |
| analyzer | 分析・パターン発見 |
| checker | 品質検査 |
```

エスカレーションを受けたとき、部長は原因を分析して、該当する社員に再実行の指示を出します。このとき、社員には「こういう条件で、ここだけ再実行しろ」と具体的に指示できます。

### 2. KPI確認と方針決定

部長は数字を見て方針を決めます。

```markdown
## KPIの構造

【主KPI】月間アフィリエイト収入（円）
  ↑ 社長が月1回確認して部長に伝える

【先行指標KPI】毎週自動計測
  KPI-1: 収益直結ページの月間クリック数
  KPI-2: 収益直結ページの平均CTR
  KPI-3: ランキングページの平均掲載順位
```

「主KPI（売上）を上げるために、先行指標（クリック数・CTR）を改善する」という因果関係を部長が理解しているので、「CTRが低いページのtitle改善を優先しろ」という判断ができます。

### 3. 失敗時のリカバリー判断

これが部長の最大の価値です。

```markdown
## 対処パターン

### パターン1: データ収集の一部失敗

判断フロー:
  データBは取得できているか？
    ├── YES → データBのみで分析を続行
    │         （analyzerに「データAなし」を伝えて実行）
    └── NO  → 全データ取得失敗 → 社長へ報告
```

**判断フローを明文化**するのがコツです。「状況を見て適切に判断して」と曖昧に書くと、AIは無限に迷うか、的外れな判断をします。

「こういう状況なら→こうする」というif-then形式で、想定される障害パターンを全て書き出してください。最初から完璧でなくていい。運用しながら「こういうケースもあったのか」とパターンを追加していけばOKです。

### 4. 社長への報告

部長でも解決できない場合にだけ、社長（人間）に報告します。

```markdown
## 社長への報告フォーマット

🚨 部長報告: {問題の要約}

状況:
  {何が起きているか}

試みた対処:
  1. {対処1} → {結果}
  2. {対処2} → {結果}

根本原因（推定）:
  {なぜ解決できないか}

お願いしたい対応:
  {具体的な手順。1〜3ステップ以内で簡潔に}
```

ここで重要なのは **「お願いしたい対応」を具体的に書かせる** ことです。「問題が起きました、どうしましょう？」という丸投げ報告は却下。「APIキーを再発行して、以下のコマンドで設定してください」と、社長が5分で対応できる形にまとめさせます。

## コマンド体系の設計

部長スキルには**コマンド**を持たせます。社長が部長に指示を出すときのインターフェースです。

### 基本コマンド

```
/bucho post    → 投稿バッチの作成を指揮
/bucho review  → 週次レビューの実行
/bucho kpi     → KPIの確認・目標設定
/bucho learn   → 知識ベースの更新
```

各コマンドの実行手順をSKILL.md内に詳細に記述します。

### `/bucho post` — 投稿バッチ作成

部長が最も頻繁に使うコマンドです。SNS運用の部長を例に見てみましょう。

```markdown
### `/bucho post` — 7日分バッチ一括作成

#### 5段階パイプライン

① 部長（方針決定・テーマ配分）
  → ② writer（投稿文作成）
    → ③ checker（品質チェック）
      → ④ schedule-post（予約投稿）
        → ⑤ 部長（ログ記録・バッチ保存）
```

注目してほしいのは、パイプラインの**最初と最後に部長がいる**こと。方針を決めるのも部長、結果を記録するのも部長。社員は中間工程を担当するだけです。

### `/bucho review` — 週次レビュー

```markdown
### `/bucho review` — 週次レビュー

#### 実行手順

Step 1: data-collector にデータ取得を指示
Step 2: checker でデータ品質を検査
Step 3: analyzer に分析・パターン発見を指示
Step 4: 部長が分析結果を確認し、判断を加える
Step 5: 知識ベースを更新
Step 6: レポートを出力
Step 7: git commit & push
```

レビューは「データを見る」だけではありません。**知識ベースを更新する**（第6章で詳述）ことで、次の投稿作成が前回より賢くなります。これが自己改善ループの核心です。

### `/bucho kpi` — KPI確認

```markdown
### `/bucho kpi`

#### 実行手順

1. 最新データから現在値を計測
2. 前月実績と比較
3. 目標達成率を計算
4. 翌月目標を自動設定:
   - 達成率100%以上 → 翌月 +20%
   - 達成率80〜99% → 目標据え置き
   - 達成率80%未満 → 翌月 +10%に下方修正
5. 達成に必要なアクションを部長の判断で提示
```

目標設定を**ルール化**するのがポイントです。「順調なら攻め、停滞なら現実路線」という方針を数値で定義しておけば、部長が感覚で判断することなく、一貫した目標管理ができます。

## 実例：SNS部門部長の1週間

SNSアカウント運用を統括する部長AIの動きを、1週間で追ってみましょう。

### 月曜日：週次レビュー

部長は先週の投稿パフォーマンスデータを`data-collector`に取得させます。

```
■ 週次レビュー（2026-03-17）

パフォーマンスサマリー:
  投稿数: 14
  平均インプレッション: 5,200
  平均エンゲージメント率: 4.8%
  前週比: +12%

ベスト投稿:
  「TOEICで900点取った人の○○...」 → imp 12,000 / eng 6.2%（Sランク）
  勝因: 数字フック + 常識否定の組み合わせ

仮説検証:
  H3「疑問形フックはエンゲージメント率が高い」
  → サンプル8/10。中間結果: +18%。あと2投稿でテスト完了予定。

知識ベース更新:
  - patterns.md: 「数字+常識否定」の組み合わせを勝ちパターンに追加
  - hooks.md: 疑問形フックのステータスを「テスト中」に更新
```

analyzer が見つけたパターンを部長が**判断して**知識ベースに登録しています。「本当にこのパターンは再現可能か？ サンプル数は十分か？」という妥当性チェックが部長の仕事です。

### 水曜日：投稿バッチ作成

部長が `/bucho post` で来週分14投稿の作成を指揮します。

```
Step 1: 方針決定
  - テーマ配分: 7テーマを2投稿ずつ（前週と重複しないよう調整）
  - 仮説テスト: 14投稿中4投稿を実験枠に割り当て
    - H3（疑問形フック）のテスト残り2投稿を含める
  - Factカテゴリ: 研究系40% / 実践系30% / 比較系20% / リスト系10%

Step 2: writer に14投稿の作成を指示
  - 各投稿にテーマ・使用するFact・テンプレートを指定
  - 実験枠には仮説タグ（H3）を付与

Step 3: checker が全14投稿を品質検査
  - 2投稿がNG（文字数超過、Factの引用なし）→ writer に差し戻し
  - 修正後、全14投稿PASS

Step 4: ユーザー承認後、schedule-post で予約投稿
```

ここでのポイントは、部長が**テーマ配分と実験配分を判断している**こと。「安定投稿70%・実験投稿30%」というバランスを取り、仮説検証のためのA/Bテスト設計まで部長がやっています。社員には「この投稿を書け」と指示するだけです。

### 金曜日：中間パフォーマンス確認

```
■ 中間チェック（2026-03-19）

月曜投稿:
  朝: 「○○を知らない人が多い...」 → imp 4,800 / eng 3.9%（B）
  夕: 「○○は実は△△だった...」 → imp 7,200 / eng 5.1%（A）
  傾向: 夕方投稿の方がエンゲージメント率が高い

火曜投稿:
  朝 [H3テスト]: 「なぜ○○なのか？」 → imp 6,100 / eng 5.8%（A）
  → H3仮説の中間データ更新: サンプル9/10、平均+22%

アクション: 特になし。来週月曜のレビューで最終判定。
```

部長は**「アクションが必要か不要か」も判断します**。中間チェックで異常値がなければ「何もしない」が正解。部長が毎回何かを変えようとすると、ノイズが増えるだけです。

### この1週間で起きたこと

- 過去のデータから勝ちパターンを**発見**
- 仮説検証を**進行**
- 来週の投稿方針を**決定**
- 社員に具体的な指示を**配信**
- 知識ベースを**更新**

社長（人間）がやったことは？ **水曜日に投稿内容を承認しただけ**です。

## 部長スキル設計のチェックリスト

部長スキルを作るとき、以下の項目が含まれているか確認してください。

- [ ] **部門の目標**が明記されているか（数字で）
- [ ] **管轄する社員の一覧**があるか
- [ ] **対処パターン**が最低3つ以上あるか
- [ ] 各パターンに**判断フロー（if-then）**が書かれているか
- [ ] **社長への報告フォーマット**があるか
- [ ] 報告には「お願いしたい対応」が含まれるか（丸投げ禁止）
- [ ] **KPIの定義**があるか
- [ ] KPIの**目標設定ルール**が数値化されているか
- [ ] **コマンド一覧**（post / review / kpi等）があるか
- [ ] 各コマンドの**実行手順**が明記されているか

## 部長の暴走を防ぐ

部長スキルに大きな権限を持たせると、当然リスクも生まれます。

### 権限と制約のテーブル

```markdown
| できること | できないこと |
|---|---|
| 配下社員の呼び出し・再指示 | 他部門への干渉 |
| 分析・実行の判断 | ユーザーのアカウント設定変更 |
| 代替手段での続行判断 | API設定・権限の変更 |
| リポジトリのファイル変更 | 社長への無断アクション |
| スキルファイルの改善提案 | — |
```

「できること」と「できないこと」を**明示的に書く**のが重要です。特に「社長への無断アクション」の禁止は絶対です。部長が「よかれと思って」勝手にデプロイしたり、外部サービスに何かを送信したりするのを防ぎます。

### エスカレーションの強制

部長がどれだけ賢くても、**解決できない問題は必ず存在します**。そのとき黙って止まるのではなく、社長に報告させる仕組みが必要です。

```markdown
## 社長へ報告すべき場合（必ず守ること）

- 全データソースが取得不能
- gitリポジトリが破損・コンフリクト解消不能
- スキル自体の根本的な設計ミス
- 2回のリカバリー試行でも解決しない問題
```

「報告すべき場合」を列挙しておけば、部長が「自分で解決しなきゃ」と延々リトライし続ける事態を防げます。

---

## まとめ

- **社員は手順を実行する。部長は状況を判断する。** この違いが組織を自律的にする
- 部長に持たせる権限は4つ：**社員の指揮、KPI管理、リカバリー判断、社長への報告**
- 対処パターンは**if-then形式で明文化**する。「適切に判断して」はNG
- コマンド体系（`/bucho post` `/bucho review` `/bucho kpi`）で社長とのインターフェースを定義
- 部長の暴走防止には**権限テーブル**と**強制エスカレーション条件**を設定

次の章では、部長が判断の根拠にする**「知識ベース」**の育て方を解説します。部長が賢くなり続ける仕組みです。
