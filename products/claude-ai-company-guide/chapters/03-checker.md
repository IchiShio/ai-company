# 第3章：品質ゲートを作る — チェッカースキル

前章で最初のスキルを作りました。記事ライターに仕事を頼むと、それらしい記事が返ってくる。

「おお、これはすごい！」

...と最初は思います。でも、しばらく運用すると気づくんです。

**AIは、平気でサボります。**

---

## AIが「やりました」と嘘をつく実例

これは実際に起きたことです。

ある記事ライタースキルに「3,000文字以上の記事を書いて」と依頼しました。返ってきた記事には「文字数: 3,200文字」と書いてありました。

実際に数えてみたら **1,800文字** でした。

別のケースでは、「データを収集して10件以上のCSVを作成して」と依頼。「✅ 完了しました。12件のデータを収集しました」と報告が来ました。

CSVを開いたら **3件** しかありませんでした。

これはAIが悪意を持って嘘をついているわけではありません。LLMの特性として、「期待されている回答を生成しようとする」傾向があるだけです。でも、結果として品質は崩壊します。

人間の組織でも同じですよね。レポートを出した本人に「このレポート、ちゃんとできてる？」と聞いたら「はい、大丈夫です」と言うに決まっています。だから組織には**レビュアー**が必要なんです。

AI会社でも同じ。**書いた人とチェックする人は、必ず分ける。**

これがチェッカースキルの存在理由です。

---

## チェッカースキルの設計パターン

チェッカースキルの構造はシンプルです。

```
入力: 前工程のスキルが出力したもの
  ↓
検査: 明確な基準に照らして1項目ずつチェック
  ↓
判定: PASS または FAIL + 具体的な理由
  ↓
FAIL時: 差し戻し or エスカレーション
```

重要なのは、**チェック項目が「客観的で曖昧さがない」こと**です。

ダメな例：
- ❌「記事の品質が高いか」→ 何をもって「高い」とするの？
- ❌「読みやすいか」→ 主観的すぎて判定できない

良い例：
- ✅「文字数が3,000文字以上か」→ 数えれば分かる
- ✅「H2見出しが5個以上あるか」→ 数えれば分かる
- ✅「根拠のない数字（9割、99%等）を含んでいないか」→ 検索すれば分かる

チェッカーに主観的な判断をさせると、チェッカー自身が「まあ大丈夫でしょう」と甘く通してしまいます。**数えられるもの、検索できるもの、照合できるもの**だけをチェック項目にするのがコツです。

---

## 実例：記事チェッカーを作る

前章の記事ライターに対応するチェッカーを作りましょう。

```markdown
---
name: article-checker
description: |
  記事ライターの出力が品質基準を満たしているか検査するスキル。
  NGなら差し戻し、2回連続NGならユーザーにエスカレーションする。

  次のような依頼で使用すること：
  - 「記事をチェックして」「品質を検査して」
  - パイプラインの品質ゲートとして自動呼び出し
---

# 記事チェッカー

## 役割

記事ライター（article-writer）の出力が品質基準を満たしているか検査する。
基準未達ならリトライを指示し、2回連続NGならユーザーにエスカレーションする。

---

## 検査項目

| # | チェック項目 | 合格基準 | 重大度 |
|---|---|---|---|
| 1 | 文字数 | 指定文字数の±10%以内 | FAIL |
| 2 | H2見出し数 | 5個以上 | FAIL |
| 3 | 各セクション文字数 | 各H2セクション300文字以上 | WARNING |
| 4 | 具体例・データ | 各セクションに最低1つ | WARNING |
| 5 | 根拠なし数字 | 「9割」「99%」等の出典なし数字がない | FAIL |
| 6 | タイトルとの整合性 | タイトルの約束が本文で果たされている | WARNING |
| 7 | 禁止事項 | ライターの禁止事項に違反していない | FAIL |

---

## 判定ルール

- **FAIL が1つでもある** → NG（差し戻し）
- **WARNING が3つ以上** → NG（差し戻し）
- **WARNING が2つ以下** → PASS

---

## PASS時の出力

```
✅ 記事チェック PASS

検査対象: article-writer
結果: PASS
WARNING: {あれば列挙、なければ「なし」}
```

---

## NG時の出力（1回目）

```
❌ 記事チェック NG

不合格項目:
  - #1 文字数: 1,800文字（基準: 3,000±10% = 2,700〜3,300文字）
  - #5 根拠なし数字: 「9割の人が挫折する」に出典なし

修正指示:
  - 文字数を3,000文字以上に増量すること
  - 「9割」の根拠を示すか、数字を使わない表現に書き換えること
```

---

## NG時の出力（2回連続）

```
🚨 エスカレーション: 記事チェック 2回連続NG

問題:
  - 文字数が依然として2,100文字（基準: 2,700〜3,300文字）

1回目の差し戻し内容:
  - 文字数を3,000文字以上に増量するよう指示

2回目の結果:
  - 300文字増えたのみ。各セクションの増量が不十分

推定原因:
  - テーマに対する情報収集が不足している可能性
```

---

## 注意事項

- 品質基準は厳格に適用する。「だいたいOK」は認めない
- チェックは客観的な基準に基づく。主観的な「良し悪し」で判定しない
- WARNING は記録しておき、傾向があればユーザーに報告する
```

---

## 重大度の設計：FAILとWARNINGを使い分ける

チェック項目には**重大度**を設定します。これが地味に重要です。

**FAIL** は「これがダメなら絶対に通さない」項目。文字数、事実の正確性、禁止事項違反など。1つでもFAILがあれば即NG。

**WARNING** は「望ましくないが、致命的ではない」項目。構成の改善余地、具体例の不足など。WARNINGが少数なら通すが、積み重なったらNG。

この使い分けがないと、些細な問題で毎回差し戻しが発生して業務が止まります。逆にすべてWARNINGにすると、重大な問題を見逃します。

判定ルールの例：
- FAIL 1つ以上 → NG
- WARNING 3つ以上 → NG
- WARNING 2つ以下 → PASS（WARNINGの内容は報告）

---

## 2回連続FAIL → エスカレーション

チェッカーの最も重要なルールが、**2回連続FAILでエスカレーション**です。

なぜ2回なのか。

1回目のFAILは「うっかりミス」かもしれません。修正指示を出せば直る可能性が高い。だから差し戻してリトライさせます。

でも、同じ問題が2回連続で直らないなら、**そのスキルの能力では解決できない問題**です。3回、4回とリトライさせても無駄にトークンを消費するだけ。

この時点で上位（部長スキル、またはユーザー本人）にエスカレーションします。

```
1回目NG → 具体的な修正指示付きで差し戻し
2回目NG → 上位にエスカレーション（問題・試行内容・推定原因を報告）
```

エスカレーションのフォーマットには必ず以下を含めます：

1. **何が問題か**（具体的なチェック項目と数値）
2. **1回目に何を指示したか**（どんな修正を求めたか）
3. **なぜ改善できなかったか**（推定原因）

これがないと、エスカレーションを受けた側が状況を把握できません。「ダメでした」だけでは困ります。人間の組織と同じで、報告には「事実・経緯・推定原因」の3点セットが必要です。

---

## チェッカーの配置：パイプラインのどこに入れるか

チェッカーは「工程の間」に挟みます。

```
記事ライター → [記事チェッカー] → 公開
                    ↓NG
               差し戻し（1回目）
                    ↓2回連続NG
               エスカレーション
```

もっと複雑なパイプラインでは、各ステップの後ろにチェッカーを入れます：

```
データ収集 → [Check 1] → 分析 → [Check 2] → 実行 → [Check 3]
```

こうすると、問題が早期に発見されます。データ収集が不完全なまま分析に進んでも、ゴミからは何も生まれません。**各工程の出口にゲートを置く**のが鉄則です。

## 複数スキルをチェックする統合チェッカー

スキルが増えてくると、スキルごとに個別のチェッカーを作るのは大変です。実用的なパターンとして、**1つのチェッカーで複数スキルの出力をチェックする**設計があります。

```markdown
## Check 1: ライターの出力チェック

| # | チェック項目 | 基準 | 重大度 |
|---|---|---|---|
| 1-1 | 文字数 | 指定範囲内 | FAIL |
| 1-2 | 構成 | H2が5個以上 | FAIL |
| 1-3 | 事実確認 | 数値に根拠あり | FAIL |

## Check 2: データ収集の出力チェック

| # | チェック項目 | 基準 | 重大度 |
|---|---|---|---|
| 2-1 | ファイル形式 | CSVヘッダーが正しい | FAIL |
| 2-2 | データ件数 | 最低10件以上 | FAIL |
| 2-3 | 数値の妥当性 | 範囲内の値 | WARNING |

## Check 3: 実行結果の出力チェック

| # | チェック項目 | 基準 | 重大度 |
|---|---|---|---|
| 3-1 | コミット確認 | 今日のコミットが存在する | FAIL |
| 3-2 | 変更件数 | 指示された件数と一致 | FAIL |
| 3-3 | 品質基準 | ファイルサイズが変更前以上 | FAIL |
```

チェック番号をセクション別に振ることで（1-1, 1-2... / 2-1, 2-2...）、どのスキルのどの項目が問題だったかを一意に特定できます。

## チェッカーが「手を抜く」問題への対処

ここで1つ問題があります。チェッカー自身もAIなので、**チェッカーが甘くチェックする**可能性があります。

これに対する実用的な対策は3つ。

### 1. チェック項目を「数値化」する

「品質が高い」→ 判定できない
「3,000文字以上」→ 数えれば判定できる

チェッカーに判断の余地を与えないことが最善策です。

### 2. 検証コマンドを書く

可能なら、チェック項目をシェルコマンドやスクリプトで検証させます。

```markdown
### 検証方法
文字数の確認:
  - 出力テキストの文字数を実際にカウントする
  - ライターの自己申告値と比較する
  - 差が10%以上あれば自己申告の信頼性にFLAG

コミットの確認:
  git log --oneline -1 で今日のコミットを確認
```

AIの自己申告ではなく、**実際のファイルやデータを読んで検証する**手順を書くと精度が上がります。

### 3. チェッカーの結果を人間がたまに抜き打ち確認する

最終的な品質保証は人間（社長）が担います。毎回見る必要はありませんが、月に1回くらいチェッカーのPASS判定を抜き打ちで確認すると、チェッカーの精度も把握できます。

---

## まとめ：チェッカーなしの運用は「目をつぶって走る」のと同じ

この章で伝えたかったことは1つだけです。

**スキルを作ったら、必ずチェッカーも作る。**

ライターだけ雇ってチェッカーなしで運用すると、品質は確実に崩壊します。最初は問題ないように見えても、10回、20回と繰り返すうちに「AIなりの手の抜き方」が蓄積していきます。

ライターとチェッカーはペアです。社員を1人雇ったら、その仕事を検査する人も同時に雇う。これがAI会社運用の基本中の基本です。

次章では、このライター＋チェッカーのペアを**パイプライン**として連結し、自動で流れる仕組みを作ります。
