# 第1章：AI会社とは何か — 設計思想

## スキル = 社員、SKILL.md = 職務記述書

Claude Codeには「スキル」という仕組みがあります。

`~/.claude/skills/` というフォルダの中に、`SKILL.md` というMarkdownファイルを置く。たったこれだけで、Claude Codeに「新しい能力」を追加できます。

```
~/.claude/skills/
├── seo-data-collector/
│   └── SKILL.md    ← SEOデータを自動収集する「社員」
├── seo-analyzer/
│   └── SKILL.md    ← データを分析して改善案を出す「社員」
├── seo-checker/
│   └── SKILL.md    ← 品質を検査する「社員」
└── x-writer/
    └── SKILL.md    ← X投稿を書く「社員」
```

ここで発想を転換します。

**スキルを「便利なテンプレート」ではなく「社員」として設計する。**

SKILL.mdは単なるプロンプトのテンプレートではありません。**職務記述書（Job Description）** です。この社員は何を担当し、何を入力として受け取り、何を出力し、どんな品質基準で仕事をし、困ったときに誰に相談するのか——それを明確に書いたドキュメントです。

実際のSKILL.mdの構造を見てみましょう。

```yaml
---
name: seo-data-collector
description: |
  自社サイトのSEOデータを3つのソースから自動収集し、
  Google Driveに日付別CSVとして保存する。データ収集のみ。
  分析は別スキルが担当。
---
```

注目してほしいのは、最後の一文です。「**分析は別スキルが担当。**」

この一文が、「万能プロンプト」と「組織」を分ける境界線です。

---

## なぜ1つの万能プロンプトではダメなのか

最初に思いつくのは、こういうアプローチです。

> 「SEOデータを収集して、分析して、改善施策を考えて、実行して、結果をチェックして、問題があったら直して、レポートにまとめて。」

1つのプロンプトに全部詰め込む。最初のうちは、これでも動きます。Claudeは賢いので、なんとなくいい感じにやってくれます。

でも、これが破綻する理由は3つあります。

### 理由1：責務が曖昧になると、品質が下がる

人間の会社でも同じです。「営業も経理も採用も全部やって」と言われた社員は、すべてが中途半端になります。

AIも同じで、1つのプロンプトに5つの仕事を詰め込むと、どの仕事にも100%の注意を払えなくなります。データ収集で手を抜いたり、分析が浅くなったり、チェックが甘くなったり。

**1スキル = 1責務**にすると、そのスキルは自分の仕事だけに集中できます。データ収集スキルは、データの取得漏れがないことだけを気にすればいい。分析スキルは、7つのフレームワークで深く分析することだけに集中できる。

### 理由2：パイプライン化できない

全部を1つのプロンプトでやると、途中で止められません。

「データ収集まではOKだけど、分析の品質がイマイチだからやり直して」ということができない。全部をもう一度最初からやり直すしかない。

スキルを分けておけば、パイプラインとして組めます。

```
データ収集 → [品質チェック] → 分析 → [品質チェック] → 実行 → [品質チェック]
```

各ステップの間にチェックポイントを置ける。問題があったステップだけやり直せる。これは1つのプロンプトでは絶対にできないことです。

### 理由3：改善・差し替えができない

万能プロンプトを改善しようとすると、全体を見直す必要があります。データ収集の部分を直したら、分析の部分に影響が出た——こういうことが頻繁に起きます。

スキルを分けておけば、1つのスキルだけを改善・差し替えできます。「分析の精度が低いから、analyzerのSKILL.mdだけ書き直そう」ということが可能になります。他のスキルには一切影響しません。

ソフトウェア開発の世界では、これを「疎結合」と呼びます。難しい概念ではありません。**それぞれの部品が独立しているから、1つだけ直せる**——ただそれだけのことです。

## 組織の3層構造

AI会社の組織は3つの層で構成されます。

```
┌─────────────────────────────────────────┐
│              社長（人間＝あなた）              │
│    最終判断 / 方針決定 / 例外対応           │
└────────────────────┬────────────────────┘
                     │ エスカレーション
                     │ （部長でも解決できないとき）
┌────────────────────▼────────────────────┐
│            COO（最高執行責任者）              │
│    部門横断の調整 / KPI管理 / 統合レポート    │
└───────┬─────────────────────┬───────────┘
        │                     │
┌───────▼───────┐     ┌───────▼───────┐
│   部長（判断者）  │     │   部長（判断者）  │
│   SEO部門       │     │    X部門        │
│  状況判断・指示   │     │  状況判断・指示   │
└───┬───┬───┬───┘     └───┬───┬───┬───┘
    │   │   │             │   │   │
   社員 社員 社員          社員 社員 社員
 collector              writer
 analyzer              checker
 executor              scheduler
 checker               analyzer
```

それぞれの役割を見ていきましょう。

### 第1層：社員スキル（実行者）

社員スキルは、決められた手順を忠実に実行します。

- **collector**: データを集める
- **analyzer**: データを分析する
- **executor**: 改善施策を実行する
- **writer**: 文章を書く
- **checker**: 品質を検査する
- **scheduler**: 予約投稿する

社員の特徴は「判断しない」ことです。SKILL.mdに書かれた手順に従って、与えられた仕事をきっちりこなす。想定外のことが起きたら、自分で判断せずに上（チェッカーや部長）に報告します。

### 第2層：部長スキル（判断者）

部長スキルは、社員を束ねて判断します。

僕の場合、以下の2人の部長がいます。

- **seo-bucho**: SEO部門の部長。collector、analyzer、executor、checkerを指揮する
- **x-bucho**: X部門の部長。writer、checker、scheduler、analyzerを指揮する

部長の仕事は3つです。

**1. 社員への指示出し**

部長は状況を見て、どの社員にどんな仕事を振るかを判断します。

```
「今週のデータを見て、CTRが低いページが5つある。
analyzer、このデータでTop 5アクションを出してくれ。」
```

**2. 失敗時のリカバリー判断**

チェッカーが「FAIL」を出したとき、部長が対処方法を判断します。

```
「analyzerの分析品質が不足しているな。
具体的なページ名と数値を入れて再分析するよう指示し直そう。」
```

**3. 解決不能時のエスカレーション**

部長でも解決できない問題だけを、社長（人間）に報告します。

```
「Search Console APIが3日連続でエラーを返している。
APIの認証情報の更新が必要かもしれません。社長、確認をお願いします。」
```

### 第3層：COO（統括者）

COOは複数の部門を横断的に管理します。

- 各部門のKPIを集約して全社ダッシュボードを作る
- 部門間の優先度を調整する（「今週はSEOより投稿を優先」等）
- API使用量やコストを追跡する
- 社長への統合レポートを作成する

小さく始めるなら、最初はCOOは不要です。部門が2つ以上になったときに初めて必要になります。

## エスカレーション設計——人間が動くのは「例外中の例外」

この組織で最も重要な設計が「エスカレーション」です。

問題が起きたとき、以下の順序で解決を試みます。

```
Step 1: 社員が自分で対処を試みる
   ↓ （失敗）
Step 2: チェッカーが検知して差し戻す
   ↓ （2回連続FAIL）
Step 3: 部長がリカバリー方法を判断する
   ↓ （部長でも解決できない）
Step 4: 社長（人間）に報告する
```

具体例を見てみましょう。

### 例：SEOパイプラインでの障害

```
1. collector がGA4のデータ取得に失敗した
   → collector が3回リトライする（社員レベルの自動リカバリー）

2. 3回とも失敗した
   → checker が「❌ FAIL: GA4データ欠損」を報告
   → パイプラインが中断される

3. 部長（seo-bucho）がエスカレーションを受け取る
   → 「GA4なしでもSearch ConsoleとAhrefsのデータで分析できるか？」を判断
   → 可能と判断 → analyzerに「GA4抜きで分析続行」を指示

4. もし部長でも対処不能な場合（APIの認証切れ等）
   → 社長に構造化レポートで報告
   → 「GA4 API認証が切れています。再認証が必要です。」
```

ポイントは、**Step 4に到達することはほとんどない**ということです。

大抵の問題はStep 1〜3で解決します。人間である僕が介入するのは、APIキーの更新やサービスの契約変更など、物理的にAIには不可能な作業だけです。

### 「2回連続FAILルール」

チェッカーの設計で特に重要なのが「2回連続FAIL」のルールです。

```
1回目のFAIL → 社員に差し戻してやり直させる
2回目のFAIL → 同じ問題が再発した → 社員レベルでは解決不能と判断
             → 部長にエスカレーション
```

これがないと、社員とチェッカーが永遠にFAIL→やり直し→FAILのループに入ります。人間の組織でも「2回やってダメなら上に相談しろ」というルールがありますよね。それと同じです。

## なぜ「組織」なのか——メンタルモデルの力

ここまで読んで、「これって要するにマイクロサービスアーキテクチャじゃないの？」と思ったエンジニアの方もいるかもしれません。

技術的には近いです。でも「組織」というメタファーを使うことには大きな意味があります。

### 設計が直感的になる

「このAPIレスポンスをパースして次のマイクロサービスに渡す」と考えるより、「この社員が仕事を終えたら、次の社員に引き継ぐ」と考えるほうが、圧倒的に設計しやすい。

「エラーハンドリングをどう設計するか」より、「この社員が困ったとき、誰に相談させるか」と考えたほうが、自然に良い設計が生まれます。

### AIへの指示が明確になる

SKILL.mdに「あなたはSEOデータを収集する社員です。分析は別の社員が担当するので、あなたはデータの正確な取得だけに集中してください」と書く。

これだけで、AIは自分の役割を正確に理解します。余計なことをしない。手を抜かない。なぜなら、自分の仕事の範囲が明確だからです。

### スケールしやすい

新しい業務が増えたら、新しい社員（スキル）を雇えばいい。チームが大きくなったら、部長を増やせばいい。とてもシンプルです。

実際、僕の組織は最初は3人の社員から始まりました。

```
初期: collector → analyzer → executor（3人）
```

今では25個以上のスキルが稼働しています。でも設計の基本は変わっていません。「1スキル = 1責務」「チェッカーで品質を保証」「困ったらエスカレーション」——この3つだけです。

---

## まとめ：AI会社の4原則

この章で解説した設計思想を、4つの原則にまとめます。

### 原則1：1スキル = 1責務
1つのスキルには1つの仕事だけを任せる。複数の仕事を詰め込まない。

### 原則2：品質はチェッカーが保証する
実行者と検査者を分ける。自分で自分の仕事をチェックさせない。

### 原則3：判断は上位層に委ねる
社員は実行に集中し、判断が必要な場面では部長に委ねる。部長でもダメなら社長に。

### 原則4：人間は例外処理だけ
日常業務はAI組織が回す。人間が動くのは、AIには物理的に不可能な作業だけ。
